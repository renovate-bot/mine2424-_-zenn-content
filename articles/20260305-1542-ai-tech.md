## 「プロンプトで頑張る」の限界

LLMを活用するとき、最初の選択肢はプロンプトエンジニアリングだ。Few-shot例を充実させ、システムプロンプトを丁寧に書き、チェーン・オブ・ソートを組み込む。多くのユースケースはこれで十分に動く。

しかしプロンプトで解決できない課題がある。特定ドメインの専門語彙に対してモデルが安定して反応しない、出力フォーマットを徹底させることが難しい、大量のコンテキストをプロンプトに詰め込むコストが高い——こうした状況になったとき、ファインチューニングが選択肢として浮上する。

ファインチューニングとは何か。モデルのウェイト（重み）を更新することだ。プロンプトがモデルの振る舞いを「入力の与え方で誘導する」のに対して、ファインチューニングは「モデルそのものを書き換える」。この違いが設計上のトレードオフを生む。

## ファインチューニングが変えるもの

LLMは膨大なテキストから「次のトークンを予測する」という事前学習によって一般的な言語能力を獲得している。ファインチューニングはこの一般能力を土台に、特定の方向性を強化する作業だ。

具体的に何が変わるかを3つの観点で整理する。

**スタイルと語調**: 医療記録を生成するモデル、法律文書を要約するモデル、カスタマーサポートの定型文を生成するモデル——それぞれに求められる文体は異なる。プロンプトに「医療専門家向けに書け」と指示するよりも、実際の医療文書で学習させた方が安定した語調になる。

**ドメイン固有の知識パターン**: 事前学習に含まれていないプロプライエタリな情報（社内規程、製品仕様書、業界固有の計算ロジック）をモデルに取り込める。ただし「知識を記憶させる」ことに対してファインチューニングは効率が悪く、RAGの方が適している場合も多い。

**タスク固有の出力形式**: 常に特定のJSON構造で返す、必ず要約を先頭に置く、コードと説明を決まった順番で出力する——こうした一貫した形式への準拠はファインチューニングで強化できる。

## 何を変えないか——ファインチューニングの誤解

ファインチューニングで「できること」と「できないこと」の混同が多い。代表的な誤解を整理しておく。

**事前学習の知識は基本的に変わらない**: ファインチューニングは事前学習で獲得した言語表現能力の上に乗る。世界知識を根本から書き換えることはできない。「2024年以降の出来事を知らせる」目的にファインチューニングを使おうとしても、RAGほどの精度は出ない。

**幻覚（ハルシネーション）は完全には解消されない**: ファインチューニングで特定の応答パターンを強化できるが、LLMが本質的に持つ「確信を持って嘘をつく」性質は消えない。重要な事実確認タスクでは、ファインチューニング後もRAGや検索ツールとの組み合わせが必要だ。

**全パラメータを更新するわけではない**: 後述するLoRAのような手法では、全体のごく一部だけを更新することで計算コストを下げる。何を更新するかによって効果の出方が変わる。

## フルファインチューニングとPEFT

ファインチューニングの手法は大きく「フルファインチューニング」と「PEFT（Parameter-Efficient Fine-Tuning）」に分かれる。

**フルファインチューニング**: すべてのパラメータを更新する。最も高い精度を出せる可能性があるが、GPUメモリと計算コストが膨大だ。70億パラメータのモデルを更新するだけでも、最新のGPU複数台が必要になる。

**LoRA（Low-Rank Adaptation）**: PEFTの代表的手法。モデルの重み行列を低ランク行列の積で近似し、その低ランク行列のみを学習する。更新するパラメータ数が全体の0.1〜1%程度に抑えられるため、メモリ効率が大幅に改善する。精度もフルファインチューニングに近い結果が得られることが多く、実務ではLoRAが主流になっている。

**QLoRA**: LoRAに量子化を組み合わせた手法。4bitに量子化したベースモデルの上でLoRAを適用することで、コンシューマーGPU（VRAM 16GB程度）でも比較的大きなモデルのファインチューニングが可能になる。

```python
# LoRAの設定イメージ（PEFTライブラリ）
from peft import LoraConfig, get_peft_model

lora_config = LoraConfig(
    r=16,              # ランク（小さいほどパラメータが少ない）
    lora_alpha=32,     # スケーリング係数
    target_modules=["q_proj", "v_proj"],  # 適用するレイヤー
    lora_dropout=0.05,
)
model = get_peft_model(base_model, lora_config)
```

ランク `r` はLoRAの表現力と計算コストのトレードオフを制御する。小さいタスクには `r=4〜16`、複雑なタスクには `r=32〜64` から試すのが一般的だ。

## データ設計がファインチューニングの品質を決める

ファインチューニングで最も重要で、かつ最も見落とされやすいのがデータ設計だ。「どんなデータで学習させるか」が結果のほぼすべてを決める。

**データ形式**: Instruction tuning（指示チューニング）では、入力と期待出力のペアを大量に用意する。単純にQAペアを並べるだけでなく、「どんな指示のバリエーションがあるか」「エッジケースをカバーできているか」を設計する。

**データ品質 vs 量**: 低品質なデータを大量に学習させても精度は上がらない。むしろ意図しない挙動が強化される。高品質なデータ1,000件がノイズ混じりの10,000件を上回ることがある。

**分布の設計**: ファインチューニングデータの分布は、本番で扱うクエリの分布に近いほど良い。本番で来るクエリを事前にサンプリングし、それに近いパターンをデータに含める。

## ファインチューニングを使うべき判断基準

RAG、プロンプトエンジニアリング、ファインチューニングのどれを選ぶかは、問題の性質によって変わる。ファインチューニングが有効なのは次のような条件が揃ったときだ。

- 同じタスクを大量（1日数百〜数千リクエスト以上）に処理する
- 出力フォーマットや語調の一貫性が品質の核心だ
- 推論コストを下げる必要がある（小さいモデルで大きいモデルに近い精度を出す）
- プロンプトだけでは安定した品質が出ない問題を特定している

逆に、知識の最新性が重要なケース、ユースケースが多様でデータ収集が難しいケース、まだ要件が固まっていないプロダクトの初期フェーズではプロンプトエンジニアリングとRAGから始める方が適切だ。

## まとめ

ファインチューニングはLLMの「振る舞い」を変える技術だ。プロンプトがモデルを誘導するのに対して、ファインチューニングはモデルそのものを書き換える。LoRAやQLoRAによってその敷居は下がったが、最大の成否はデータ設計にある。「何をファインチューニングで解決すべきか」を見極める判断力と、高品質なデータを用意する能力がファインチューニング活用の本質的なスキルだ。
