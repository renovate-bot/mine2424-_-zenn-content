## はじめに

FlutterとFirebaseの組み合わせはモバイルアプリ開発の定番だ。`flutterfire_cli` でセットアップし、`cloud_firestore` パッケージを追加すれば数時間でデータの読み書きができる。

しかし「動くコード」と「設計として正しいコード」の間には大きな差がある。Firestoreのクエリをウィジェットに直接書いたり、FirebaseAuthのユーザー状態をアプリ全体に散らばったりしていないか。Firebaseを「正しく統合する」ための設計思想を整理する。

## FirebaseをRepository層で抽象化する

最も重要な設計判断は、**FirebaseのSDKを直接ウィジェットやProviderから呼ばない**ことだ。

Firestoreを例に取ると、`FirebaseFirestore.instance.collection('users').doc(uid).get()` というコードをUIレイヤーに書いてしまうと、次の問題が起きる。テストが書けない（Firestoreは実際のインフラに依存する）、FirestoreをSQLiteやAPIに変更したとき影響範囲が広い、エラーハンドリングが散在する。

Repository パターンで抽象化する。

```dart
abstract class UserRepository {
  Future<User?> findById(String uid);
  Future<void> save(User user);
  Stream<User?> watchById(String uid);
}

class FirestoreUserRepository implements UserRepository {
  final FirebaseFirestore _firestore;
  
  @override
  Stream<User?> watchById(String uid) {
    return _firestore
      .collection('users')
      .doc(uid)
      .snapshots()
      .map((snap) => snap.exists ? User.fromJson(snap.data()!) : null);
  }
}
```

ウィジェットやRiverpodのProviderは `UserRepository` インターフェースに依存する。テスト時は `MockUserRepository` を差し込める。Firestoreがバックエンドである事実はUIレイヤーから完全に隠蔽される。

## FirebaseAuthの状態管理

認証状態の管理はアプリ全体に影響するため、設計を誤ると後から修正が困難になる。

`FirebaseAuth.instance.authStateChanges()` はStreamを返す。これをRiverpodの `StreamProvider` でラップするのが標準的なアプローチだ。ログイン・ログアウト・トークン更新が自動的に反映される。

重要なのは**認証状態とユーザーデータを分けて管理する**ことだ。`FirebaseUser`（FirebaseAuthのモデル）と、アプリのドメインモデルとしての `User`（名前・プロフィール・設定など）は別物だ。認証状態が変わるたびにFirestoreからユーザープロフィールを取得するウォッチャーを設け、アプリ内では常にドメインモデルの `User` を参照する。

また、ルーティングとの統合も重要だ。`go_router` の `redirect` に認証状態のチェックを入れ、未ログインのユーザーがプロテクトされたルートにアクセスした場合は自動的にログイン画面にリダイレクトする。この処理をウィジェット内の `initState` に書くのは設計として正しくない。

## FirestoreのデータモデルとN+1問題

FirestoreはSQLと異なり、JOINが存在しない。コレクション構造とクエリは表裏一体に設計する必要がある。

典型的な失敗パターンが「一覧を取得してから各詳細を別途取得する」というN+1だ。Firestoreでは各ドキュメント取得がネットワークリクエストになる。100件のリストを表示するために101回のリクエストが走ることになる。

対策は2つある。**データの非正規化**（表示に必要な情報を一か所に持たせる）と、**`whereIn` クエリの活用**（最大30件まで一括取得）だ。

「どういうクエリを発行したいか」を先に決め、それに合わせてコレクション構造を設計する。これがFirestoreの基本思想だ。

## Crashlyticsによるクラッシュ監視の設計

Crashlyticsはクラッシュレポートを収集するが、デフォルトの設定だけでは十分な情報が得られないことがある。

設計上重要なのは**カスタムキーとカスタムログを戦略的に付与する**ことだ。ユーザーIDをセットしておくことで「このユーザーがクラッシュした」という追跡が可能になる。また、ユーザーの重要な操作（画面遷移・フォーム送信・API呼び出し）をログとして記録しておくことで、クラッシュ直前の操作列が追える。

FlutterはDartの例外とFlutterフレームワークの例外を別々に捕捉する必要がある。`FlutterError.onError`（フレームワーク例外）と `PlatformDispatcher.instance.onError`（非同期例外）の両方を設定しないと、一部のクラッシュが見逃される。

## まとめ

FlutterとFirebaseの統合で大切なのは「Firebaseを直接触るコードを端に追いやる」設計思想だ。Repository層でFirestoreを隠蔽し、StreamProviderで認証状態を管理し、Crashlyticsにコンテキストを付与する。

Firebaseは強力だが、それだけにビジネスロジックとインフラが混ざりやすい。疎結合に保つことで、スケールアップ時の変更コストとテストのしやすさが大きく変わる。
