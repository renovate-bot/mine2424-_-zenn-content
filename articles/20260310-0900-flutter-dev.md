## はじめに

FlutterとFirebaseの組み合わせは、モバイルアプリ開発の定番だ。認証、データベース、クラッシュレポート——これだけのインフラをバックエンドなしで揃えられるのは魅力的だ。

ただ「とりあえず使える」状態と「設計として正しい」状態の間には、大きな差がある。Firestoreのデータ読み書きをコンポーネントに直書きしてしまったり、認証状態の管理が散らばってしまったり——後から整理が難しくなるパターンはよく見られる。

Firebaseを「正しく」使うとはどういうことか、を設計の観点から考えてみたい。

## Firebaseを「インフラ」として扱う

最初に決めるべき原則は、**FirebaseのSDKをUIコードから隔離する**ことだ。

`FirebaseFirestore.instance.collection('users').doc(uid).get()` を画面ウィジェットに直接書いてしまうと、テストが書けなくなる。Firebaseに依存した画面のテストは難しい。また、将来的にFirestoreを別のバックエンドに置き換えたいとなったとき、変更が全画面に広がる。

Repositoryパターンを使う。

```dart
// abstract interface
abstract class UserRepository {
  Future<User?> findById(String uid);
  Future<void> save(User user);
}

// Firebase実装
class FirestoreUserRepository implements UserRepository {
  final FirebaseFirestore _db;
  
  FirestoreUserRepository(this._db);
  
  @override
  Future<User?> findById(String uid) async {
    final doc = await _db.collection('users').doc(uid).get();
    if (!doc.exists) return null;
    return User.fromJson(doc.data()!);
  }
}
```

UIはinterfaceだけを知っている。FirestoreUserRepositoryはDIで注入する。テストではモック実装を差し込める。

## Firebase Authの状態管理

認証状態は、アプリ全体で共有されるグローバルな状態だ。これをどう管理するかは、アプリ設計の根幹に関わる。

`firebase_auth`パッケージは`authStateChanges()`というStreamを提供する。ログイン・ログアウト・セッション切れが起きると、このStreamが値を流す。

Riverpodと組み合わせるのが自然だ。

```dart
@riverpod
Stream<User?> authState(AuthStateRef ref) {
  return FirebaseAuth.instance.authStateChanges();
}
```

このProviderをアプリのルートで監視して、認証状態に応じてルーティングを切り替える。go_routerの`redirect`機能と組み合わせると、「未認証ユーザーはログイン画面へ」という振る舞いを一箇所で管理できる。

## Firestoreのデータモデリング

FirestoreはNoSQLのドキュメントDBだ。RDBと違い、正規化よりも「読み取りパターンに合わせた非正規化」を意識する必要がある。

Firestoreの課金はドキュメントの読み取り回数に基づく。1回のリクエストで必要なデータをすべて取れるように設計する。JOINはない——代わりに、よく一緒に読まれるデータは同じドキュメントに埋め込む（サブコレクションか、フィールドとして）。

Dart側では、freezedを使ったモデルクラスとjson_serializableを組み合わせて、FirestoreのMapとの変換を型安全に行う。

セキュリティルールも重要だ。FirestoreはクライアントSDKから直接アクセスできる分、セキュリティルールで「誰が何を読み書きできるか」を厳密に定義する必要がある。UIのアクセス制御だけではなく、データベース側でも守る。

## Crashlyticsでクラッシュを捕捉する

`firebase_crashlytics`は、アプリのクラッシュをリアルタイムで収集する。リリース後の品質監視に欠かせない。

Flutterアプリでは、いくつかのレイヤーでエラーを捕捉する必要がある。

```dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp();
  
  // Flutterフレームワーク内のエラー
  FlutterError.onError = FirebaseCrashlytics.instance.recordFlutterFatalError;
  
  // Dartの非同期エラー
  PlatformDispatcher.instance.onError = (error, stack) {
    FirebaseCrashlytics.instance.recordError(error, stack, fatal: true);
    return true;
  };
  
  runApp(const MyApp());
}
```

Crashlyticsはユーザー識別子やカスタムキーを付与できる。「どのユーザーでクラッシュが起きたか」「クラッシュ直前にどの操作をしていたか」というコンテキストを残すことで、再現性の高いバグ修正が可能になる。

## Firebase Performanceで実測する

Crashlyticsでクラッシュを、Firebase Performanceで遅延を監視する。

アプリの起動時間、画面遷移にかかる時間、HTTPリクエストのレイテンシが自動的に計測される。カスタムトレースを追加すれば、特定の処理（Firestoreのデータ読み込みなど）にかかる時間も可視化できる。

「体感が重い」という感覚的な報告を、数値として捉えられるようになる。

## Firebaseに過度に依存しない

Firebaseは強力だが、ベンダーロックインのリスクもある。

Repositoryパターンを使ってFirebase SDKを隔離しておけば、将来的に別のバックエンドに移行することが現実的になる。特にFirestoreは独自のデータモデルがあるため、設計を誤るとFirestoreなしでは動かないアプリになってしまう。

「FlutterアプリがFirebaseを使う」のではなく、「FlutterアプリのインフラとしてFirebaseを選ぶ」という意識の差が、設計の柔軟性を保つ。

## まとめ

FlutterとFirebaseの統合で重要なのは、**Firebaseを「インフラ」として扱う**設計だ。

SDKをUIから隔離するRepositoryパターン、認証状態のStream管理、読み取りパターンに合わせたFirestoreのデータモデリング、そしてCrashlyticsによるクラッシュ監視——これらを組み合わせることで、スケールしても管理しやすいアプリの基盤を作ることができる。
