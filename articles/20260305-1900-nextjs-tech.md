## はじめに

Next.jsのEdge Runtimeという言葉を聞いて、「なんとなく速そう」「CDNに近い場所で動くやつ」という印象を持っている人は多いのではないか。

しかし、Edge Runtimeは万能ではない。むしろ**できることが大幅に制限されたランタイム**であり、その制約を理解した上で使わなければ、思わぬところで詰まることになる。

本記事では、Edge Runtimeがなぜ速いのか、どんな処理に向いていて何が苦手なのかを整理する。

---

## Edge Runtimeとは何か

通常のNext.jsアプリはNode.jsランタイムで動作する。サーバーサイドの処理はNode.jsの上で実行され、APIルートやServer Componentのレンダリングがここで行われる。

Edge Runtimeは、これとは異なるランタイム環境だ。Cloudflare WorkersやVercel Edge Networkのように、**世界中に分散したエッジノード上でコードを実行する**仕組みだ。

最大の特徴は「コールドスタートがほぼゼロ」であることと、「ユーザーの地理的に近い場所で処理が完結する」ことだ。これが「速さの正体」だ。

---

## なぜEdgeは速いのか——アーキテクチャから理解する

通常のサーバーレス関数（例：Vercel FunctionsのNode.jsランタイム）には、コールドスタート問題がある。関数が一定時間使われていないと、次のリクエスト時に「起動時間」が発生する。数百ミリ秒〜数秒のレイテンシが加わる。

Edge Runtimeがこれを回避できる理由は2つある。

1つ目は**V8 Isolateという実行モデル**だ。通常のNode.jsはプロセスベースで動作するが、V8 IsolateはChromiumのJavaScriptエンジンを直接使い、プロセスの代わりに非常に軽量な「分離実行環境」を使う。起動コストが桁違いに小さい。

2つ目は**地理的分散**だ。エッジノードはCDNのPoP（Points of Presence）のような形で世界中に配置されている。東京のユーザーへのレスポンスは、遠くの中央サーバーではなく、近くのエッジノードから返される。ネットワークレイテンシが物理的に小さくなる。

---

## Edge Runtimeの制約——何ができないのか

速さの代わりに、Edge Runtimeは多くの制約を持つ。ここを理解していないと設計が破綻する。

**Node.js APIが使えない**

Edge RuntimeはNode.jsではなく、Web Standardsベースの環境だ。つまり以下が使えない：

- `fs`（ファイルシステム）
- `child_process`
- `net`, `tls`などの低レベルネットワークAPI
- 多くのNode.js専用npmパッケージ

使えるのは`fetch`、`Request`/`Response`、`crypto`など、Web APIとして標準化されたものに限られる。

**実行時間の制限**

Edge Runtimeには厳しい実行時間制限がある（Vercelでは通常25秒、プランによって異なる）。長時間の処理、重いAI推論、大量データの変換などには向かない。

**バンドルサイズの制限**

Vercel Edge Functionsでは1MB（圧縮後）というバンドルサイズ制限がある。大きなライブラリは使えないか、工夫が必要になる。

**データベース直接接続の制限**

PostgreSQLやMySQLのような従来のデータベースは、TCP接続ベースだ。Edge環境ではTCP接続の確立に制限があるため、PrismaやknexなどのORM/クライアントがそのままでは動かないことが多い。代替として、HTTP/WS経由でアクセスできるエッジ対応のデータベース（Neon, Turso, PlanetScaleなど）や、接続プールサービスを使う必要がある。

## Next.jsでEdge Runtimeを指定する方法

Next.jsでは、Route HandlersやMiddlewareでEdge Runtimeを選択できる。

```typescript
// Route Handlerでの指定
export const runtime = 'edge';

export async function GET(request: Request) {
  return new Response('Hello from the Edge!', {
    headers: { 'Content-Type': 'text/plain' },
  });
}
```

MiddlewareはデフォルトでEdge Runtimeで実行される。

```typescript
// middleware.ts（デフォルトでEdge）
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  // 認証チェック、ヘッダー付与、A/Bテストなど
  return NextResponse.next();
}
```

## Edge Runtimeが輝くユースケース

制約を踏まえた上で、Edge Runtimeが真価を発揮するのはどんな場面か。

### 認証・認可のチェック

JWTの検証やCookieのチェックは計算量が小さく、Web Crypto APIで実装できる。すべてのリクエストに入るMiddlewareにEdgeを使えば、バックエンドに到達する前にレイテンシ最小で認証を完了できる。

### A/Bテスト・フィーチャーフラグ

ユーザーごとに異なるバリアントを返す処理は、単純な条件分岐で実装できる。コールドスタートなしに瞬時に判断し、適切なページやヘッダーを返せる。

### 地理情報を使ったルーティング

Vercel EdgeではリクエストヘッダーにGeoIPの情報が含まれる。言語切り替えや地域限定コンテンツの制御を、エッジで行える。

### 軽量なAPIプロキシ・レスポンスヘッダー操作

外部APIへのプロキシ、セキュリティヘッダーの付与、キャッシュ制御——こうした「通過する処理」はEdgeに適している。

---

## Node.jsランタイムを選ぶべき場面

逆に、以下の処理はNode.jsランタイムを使うべきだ：

- PostgreSQL/MySQLへの直接接続が必要な処理
- `fs`を使ったファイル操作（PDFの生成など）
- 実行時間がかかる重い処理（AI推論、大量データ集計）
- Node.js専用のライブラリを使っている処理
- PrismaなどのORM（Prisma Accelerateなどエッジ対応版を除く）

判断の軸はシンプルで、**「Web Standardsだけで実装できる、軽量な処理か」**——これがEdgeを選ぶ条件だ。

---

## まとめ：Edgeは「全てを速くする魔法」ではない

Edge Runtimeを使えばすべてが速くなる、という誤解は禁物だ。Edgeが速いのは「コールドスタートがない」「地理的に近い」という特性からであり、それは**軽量で制約内に収まる処理に限った話**だ。

向いている場所でEdgeを使うことで、ユーザー体験を確実に改善できる。一方で無理にEdgeに寄せようとすると、制約にぶつかって回り道することになる。

ランタイムの選択は、コードの書き方の問題ではなく**アーキテクチャの決断**だ。何を処理したいのかを明確にした上で、Node.jsとEdgeを適切に使い分けることが、Next.jsアプリの設計の肝になる。
