# AIエージェントに「計画」と「実行」を分離させると何が変わるか

## はじめに

AIエージェントを実装していると、ある共通の問題にぶつかることがある。それは「考えながら動く」構造が、思ったより脆いという問題だ。

ツールを選びながら、同時に結果を評価しながら、さらに次のステップを検討しながら——そのすべてを一つのプロンプトループで処理しようとすると、エージェントは頻繁に道に迷う。途中で目標を忘れたり、無関係なツールを呼んだり、あるいは同じことを繰り返したりする。

この問題を解決するアプローチの一つが、**計画フェーズと実行フェーズの分離**だ。この記事では、なぜ分離が有効なのか、どう設計すれば良いのかを整理する。

---

## 「考えながら動く」の何が問題か

人間でも、複雑な作業を「考えながら動く」と失敗しやすい。たとえば料理をしながら同時に献立を考え、冷蔵庫の在庫を確認し、調理手順を覚えるのは難しい。料理が上手な人は先に献立を決め、材料を揃えてから調理に集中する。

LLMも同じだ。コンテキストウィンドウの中で「次は何をするか」と「今これをどう処理するか」を同時に扱うと、どちらも中途半端になりやすい。特にツール呼び出しの結果がコンテキストに蓄積されていくと、モデルは後半になるほど最初の目標から離れた判断をしやすくなる。

これを「コンテキスト汚染」と呼ぶことがある。実行ログが計画に干渉するのだ。

---

## 計画フェーズと実行フェーズを分ける

解決策はシンプルで、**エージェントを二つの役割に分ける**ことだ。

**プランナー（Planner）**
- ユーザーの目標を受け取り、達成に必要なステップを洗い出す
- 具体的な行動リストを構造化して出力する
- ツールは呼ばない。あくまで計画だけを立てる

**エグゼキューター（Executor）**
- プランナーが出力した計画を一行ずつ実行する
- ツールを呼び、結果を確認し、次のステップに進む
- 計画を変更しない。実行に専念する

この分離によって、それぞれのモデル（あるいはプロンプト）は一つの責任だけを持てばよくなる。プランナーは全体俯瞰、エグゼキューターは局所最適——それぞれが得意なことに集中できる。

---

## 実装上の考え方

### プランナーの出力形式

プランナーには、実行可能なステップを構造化して出力させることが重要だ。自然言語の箇条書きではなく、機械的に処理できる形式が望ましい。

```json
{
  "goal": "ユーザーの質問に回答する",
  "steps": [
    { "id": 1, "action": "search", "input": "検索クエリ" },
    { "id": 2, "action": "summarize", "input": "step_1_output" },
    { "id": 3, "action": "respond", "input": "step_2_output" }
  ]
}
```

このようにJSON形式で出力させると、エグゼキューターはパースして順に実行するだけでよい。エラーが起きたときのリカバリーも、どのステップで失敗したかが明確になる。

### 計画を途中で修正するか

純粋な分離では、計画は変更しない。だが実際には途中で計画が崩れることがある。そのとき、いくつかの選択肢がある。

1. **エラーを上位に伝搬させ、プランナーに再計画させる**
2. **エグゼキューターに小さな軌道修正を許可する（ただし計画の本質は変えない）**
3. **事前に「失敗時の代替ステップ」をプランナーに作成させる**

どれが正解かはユースケース次第だが、初期は「エラー→再計画」のシンプルなループが実装しやすく、挙動も予測しやすい。

### コンテキストの隔離

計画フェーズと実行フェーズでコンテキストを共有しないことも重要だ。エグゼキューターが受け取るのは「計画書」だけ。プランナーが考えた過程（迷い、候補、棄却した選択肢）は渡さない。これにより実行フェーズのコンテキストがクリーンに保たれる。

---

## この設計がもたらす副産物

計画と実行を分離すると、予期せず得られるメリットがいくつかある。

**デバッグのしやすさ**
何か失敗したとき、「計画が悪かったのか、実行が悪かったのか」を切り分けやすい。計画書をログとして残しておけば、後から「なぜその順番で実行したのか」を追跡できる。

**ユーザーへの透明性**
計画書をユーザーに見せることができる。「これからこういう手順で作業します」と示せると、ユーザーは進捗を理解しやすく、途中での軌道修正も依頼しやすい。

**並列実行への拡張**
計画が構造化されていれば、依存関係のないステップを並列実行することも可能になる。「ステップ2とステップ3は独立しているので同時に実行する」という判断が、プランナーの出力時点でできる。

---

## どういう場合に分離が有効か

すべてのエージェントに分離が必要なわけではない。

**分離が有効な場合：**
- ステップ数が多く、全体の見通しが重要なタスク
- ツールの呼び出し回数が多く、コスト管理が必要な場合
- ユーザーに進捗を見せたい場合
- 失敗時のリカバリーを明確にしたい場合

**分離が不要な場合：**
- シンプルな1〜2ステップのタスク
- リアルタイムの対話が必要で、計画まで待てない場合
- ステップが高度に動的で、事前計画が意味をなさない場合

設計の選択は、タスクの複雑さと予測可能性によって決まる。

---

## まとめ

AIエージェントが「考えながら動く」構造には本質的な脆さがある。計画フェーズと実行フェーズを分離することで、それぞれの役割を明確にし、コンテキスト汚染を防ぎ、デバッグとユーザー体験を向上させることができる。

重要なのは、分離は「完璧な設計を目指すため」ではなく、「失敗したときに何が悪かったかを特定しやすくするため」でもあるという点だ。エージェントはまだ失敗する。その失敗から学べる構造を持つこと——それが今のAIエージェント開発に求められる実践的な設計原則だと思う。
