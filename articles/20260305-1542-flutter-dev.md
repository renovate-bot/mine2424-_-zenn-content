## ローカルDBが必要になる瞬間

Flutterアプリの状態管理にはRiverpodやBlocが活躍する。しかしアプリを閉じてもデータを保持したい、サーバーなしにオフラインで動かしたい、複雑なクエリで大量のデータを扱いたい——こうした要件になったとき、SharedPreferencesや単純なjsonファイルでは限界がある。

SQLiteはFlutterアプリのローカルDBとして長年使われてきた。`sqflite`パッケージが代表的な選択肢だが、sqfliteはSQLを文字列で書くため、タイポや型の不一致がコンパイル時に検出されない。大規模な使い方をするとクエリの管理が難しくなりやすい。

DriftはFlutter向けのSQLiteラッパーで、Dartのコードを書くことで型安全なSQLクエリを実現する。テーブル定義もDartのクラスで書き、スキーマの変更時のマイグレーションも管理できる。本記事では、Driftの設計思想と使い方を整理する。

## Driftが解決する問題

sqfliteの典型的な使い方では、次のようなコードになる。

```dart
// sqfliteでの例（文字列でSQL）
final List<Map<String, dynamic>> maps = await db.query(
  'tasks',
  where: 'is_completed = ?',
  whereArgs: [1],
);
```

`'tasks'`というテーブル名、`'is_completed'`というカラム名が文字列になっている。テーブル名を変更してもこの文字列は静的解析では検出されない。カラムの型が変わっても、キャストはランタイムまで気づかない。

Driftではテーブルをクラスとして定義し、クエリもDartのメソッドチェーンで書く。コンパイル時の型チェックが働き、テーブルのリネームやカラムの変更があれば関連するクエリのコンパイルエラーとして即座にわかる。

## テーブル定義の考え方

Driftのテーブルは`Table`を継承したDartクラスとして定義する。

```dart
// タスク管理アプリのテーブル定義例
class Tasks extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get title => text().withLength(min: 1, max: 200)();
  TextColumn get description => text().nullable()();
  BoolColumn get isCompleted => boolean().withDefault(const Constant(false))();
  DateTimeColumn get createdAt => dateTime()();
  IntColumn get categoryId => integer().nullable().references(Categories, #id)();
}

class Categories extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get name => text().withLength(min: 1, max: 100)();
  IntColumn get colorHex => integer()();
}
```

各カラムは`IntColumn`、`TextColumn`、`BoolColumn`などの型付きColumnクラスで宣言する。`nullable()`、`withDefault()`、`withLength()`といったチェーン可能な制約を付けることで、SQLのDDLとほぼ対応した形で制約を表現できる。

外部キー制約（`references()`）もDartのコードで表現でき、スキーマの関係性がコードで見える。

## DAOでクエリをカプセル化する

Driftでは「DAO（Data Access Object）」パターンでクエリをカプセル化することを推奨している。テーブルに対する操作をDAOクラスにまとめることで、データ操作の責任範囲が明確になる。

```dart
part of 'app_database.dart';

@DriftAccessor(tables: [Tasks, Categories])
class TasksDao extends DatabaseAccessor<AppDatabase> with _$TasksDaoMixin {
  TasksDao(AppDatabase db) : super(db);

  // 未完了タスクをカテゴリJoinで取得
  Stream<List<TaskWithCategory>> watchIncompleteTasks() {
    final query = select(tasks).join([
      leftOuterJoin(categories, categories.id.equalsExp(tasks.categoryId)),
    ])
      ..where(tasks.isCompleted.equals(false))
      ..orderBy([OrderingTerm.asc(tasks.createdAt)]);

    return query.watch().map((rows) =>
      rows.map((row) => TaskWithCategory(
        task: row.readTable(tasks),
        category: row.readTableOrNull(categories),
      )).toList()
    );
  }

  Future<void> completeTask(int id) =>
    (update(tasks)..where((t) => t.id.equals(id)))
      .write(const TasksCompanion(isCompleted: Value(true)));
}
```

`watch()`はStreamを返すため、データベースの変更をリアクティブに監視できる。`select()`、`where()`、`join()`はすべてDartのメソッドで、クエリをDartの型システムで表現する。

## マイグレーションの設計

アプリのバージョンアップでスキーマが変わるとき、既存のユーザーのデータを壊さずに安全に移行する仕組みが必要だ。これがマイグレーションだ。

Driftは`schemaVersion`とマイグレーションコールバックでこれを管理する。

```dart
@DriftDatabase(tables: [Tasks, Categories], daos: [TasksDao])
class AppDatabase extends _$AppDatabase {
  AppDatabase(QueryExecutor e) : super(e);

  @override
  int get schemaVersion => 2; // バージョンを上げる

  @override
  MigrationStrategy get migration => MigrationStrategy(
    onUpgrade: (migrator, from, to) async {
      if (from < 2) {
        // バージョン1→2: Categoriesテーブルを追加
        await migrator.createTable(categories);
        // 既存Tasksにcategoryカラムを追加（NULLable）
        await migrator.addColumn(tasks, tasks.categoryId);
      }
    },
  );
}
```

`schemaVersion`を上げてマイグレーションを書くことで、旧バージョンのアプリからのアップグレード時に自動でスキーマが更新される。Driftには`GeneratedDatabase`テスト用のスキーマスナップショット機能があり、マイグレーションの正しさを自動テストで検証することもできる。

## テスト可能な設計

Driftはインメモリデータベース（`NativeDatabase.memory()`）をサポートしているため、実際のSQLiteファイルを使わずにデータベースのテストを書ける。

```dart
test('未完了タスクのみが返される', () async {
  final db = AppDatabase(NativeDatabase.memory());
  final dao = db.tasksDao;

  await dao.insertTask(TasksCompanion.insert(
    title: 'テストタスク',
    isCompleted: const Value(false),
    createdAt: DateTime.now(),
  ));

  final tasks = await dao.watchIncompleteTasks().first;
  expect(tasks.length, 1);
  await db.close();
});
```

DAOにテーブル操作をカプセル化しているので、テストでDAOを直接使えば状態管理レイヤーとは独立してデータ層をテストできる。

## Riverpodとの組み合わせ

実際のアプリでは、DriftのDAOをRiverpodのProviderで提供し、UIからWatchするパターンが使いやすい。

DAOの`watch()`メソッドはStreamを返すため、RiverpodのStreamProviderとの相性がいい。DAOのStreamをStreamProviderで包めば、RiverpodのUI側でデータの変化をリアクティブに受け取れる。

データベースのProviderをオーバーライドするRiverpodの機能と組み合わせると、テスト時にインメモリDBに差し替えることも容易だ。

## sqfliteからDriftへの移行を考えるとき

既存のsqfliteコードをDriftに移行する必要はない。Driftの価値が大きくなるのは次のような条件だ。

- 複数テーブルをJoinしたクエリが増えてきた
- スキーマの変更に伴うマイグレーションを管理したい
- テーブルへのアクセスを型安全に保ちたい
- 複数人で開発していてスキーマの変更が追いにくい

小規模なアプリや、データ構造が安定していてSQLが数行のケースでは、sqfliteや`isar`（NoSQL型のDartネイティブDB）の方がシンプルに使える場合もある。Driftはスキーマの複雑さとクエリの型安全性が重要になってきたタイミングで真価を発揮する。

## まとめ

DriftはFlutterでSQLiteを扱うための型安全なアクセス層だ。テーブルをDartクラスで定義し、クエリをメソッドチェーンで書くことで、コンパイル時に型の不一致やスキーマの変化を検出できる。DAOパターンによるカプセル化、マイグレーション管理、インメモリDBを使ったテスト対応が揃っており、スキーマが複雑化してきたアプリのデータ層設計に適している。RiverpodのStreamProviderとの組み合わせで、データベースの変化をリアクティブにUIに反映する設計も自然に実現できる。
