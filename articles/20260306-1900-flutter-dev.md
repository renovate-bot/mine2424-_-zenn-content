## はじめに：アクセシビリティはアフターケアではない

Flutterアプリ開発において、アクセシビリティ対応は「最後に時間が余ったらやるもの」として扱われがちだ。しかしアクセシビリティとは、スクリーンリーダーを使うユーザーのためだけにある特別な機能ではない。それは「UIが正しく意味を持っているか」という、設計の根本的な品質に関わる問題だ。

Flutterのアクセシビリティ機構の中心にあるのが `Semantics` ウィジェットだ。このウィジェットを理解することは、視覚的な見た目の裏側に「意味の構造」を正しく構築することを意味する。

---

## Semanticsとは何か

Flutterのウィジェットツリーは視覚的なレンダリングツリーを構成するが、それとは別に「セマンティクスツリー」という構造が存在する。セマンティクスツリーはスクリーンリーダー（iOS の VoiceOver、Android の TalkBack）が参照する、UIの意味構造だ。

Flutterの多くの標準ウィジェット（`ElevatedButton`、`Text`、`TextField` など）は、デフォルトで適切なセマンティクス情報を自動的に生成する。しかしカスタムウィジェットや複雑なUIを構築すると、セマンティクスツリーが意図通りに構成されないことがある。

`Semantics` ウィジェットはセマンティクスツリーに情報を追加・上書き・調整するための仕組みだ。

```dart
Semantics(
  label: '戻るボタン',
  button: true,
  onTap: () => Navigator.pop(context),
  child: GestureDetector(
    onTap: () => Navigator.pop(context),
    child: Icon(Icons.arrow_back),
  ),
)
```

このコードでは、視覚的には矢印アイコンが表示されるが、スクリーンリーダーには「戻るボタン」というラベルと、それがボタンであるという情報が伝わる。

## セマンティクスツリーを確認する方法

実装が正しいかどうかを確認するには、実際にセマンティクスツリーを見ることが重要だ。

**Flutter Inspector**  
Flutter DevTools の Inspector を使うと、現在のウィジェットツリーのセマンティクス情報を確認できる。"Show semantics" オプションを有効にすることで、セマンティクスのオーバーレイが表示される。

**デバイスのアクセシビリティ機能**  
実機やエミュレータで VoiceOver（iOS）または TalkBack（Android）を有効にして、実際に操作してみることが最も確実な確認方法だ。スクリーンリーダーが何を読み上げるか、操作の流れが自然かどうかを体験できる。

**`debugDumpSemanticsTree()`**  
コード中で呼び出すと、現在のセマンティクスツリーをコンソールに出力できる。自動テストと組み合わせて使う場面もある。

## よくあるセマンティクスの問題と対処法

### 問題1：意味のないラベルが読まれる

カスタムアイコンボタンや、絵文字・画像だけのUIでは、スクリーンリーダーが意味のない情報を読み上げる。

```dart
// ❌ アイコンのコードポイントがそのまま読まれてしまう
IconButton(
  icon: Icon(Icons.favorite),
  onPressed: () {},
)

// ✅ tooltipまたはSemanticsで意味あるラベルを付与
IconButton(
  icon: Icon(Icons.favorite),
  tooltip: 'お気に入りに追加',
  onPressed: () {},
)
```

`IconButton` は `tooltip` プロパティを持っており、設定するだけでセマンティクスラベルになる。

### 問題2：複数の要素がバラバラに読まれる

カード型UIのように複数のウィジェットが組み合わさった要素では、スクリーンリーダーが各要素を個別にフォーカスしてしまい、全体の意味が伝わりにくくなる。

```dart
// カード全体を1つの要素としてまとめる
Semantics(
  label: '山田太郎、シニアエンジニア、東京オフィス',
  button: true,
  child: UserCard(
    name: '山田太郎',
    role: 'シニアエンジニア',
    office: '東京オフィス',
    onTap: () => showUserDetail(),
  ),
)
```

`Semantics` でグループ化することで、カード全体が1つのアクセシブルな要素として扱われる。

### 問題3：セマンティクスが不要な装飾要素にもフォーカスが当たる

ビジュアル的な装飾（背景のパターン、アイコンの影など）はセマンティクスツリーから除外すべきだ。

```dart
ExcludeSemantics(
  child: DecorativeBackground(), // 装飾的な背景
)
```

`ExcludeSemantics` を使うと、その子ウィジェット以下がセマンティクスツリーから完全に除外される。

## フォーカス順序の設計

スクリーンリーダーユーザーは通常、UI要素を順番にフォーカスしながら操作する。このフォーカスの順序が論理的でないと、使いにくいUIになる。

Flutterのデフォルトでは、ウィジェットツリーの深さ優先順にフォーカスが移動する。しかし複雑なレイアウトでは、視覚的な読み取り順序とフォーカス順序がずれることがある。

`FocusTraversalGroup` と `FocusTraversalOrder` を使うことで、フォーカス順序を明示的に制御できる。

```dart
FocusTraversalGroup(
  policy: OrderedTraversalPolicy(),
  child: Column(
    children: [
      FocusTraversalOrder(
        order: NumericFocusOrder(1),
        child: TitleText(),
      ),
      FocusTraversalOrder(
        order: NumericFocusOrder(3), // 意図的に3番目
        child: ActionButton(),
      ),
      FocusTraversalOrder(
        order: NumericFocusOrder(2),
        child: DescriptionText(),
      ),
    ],
  ),
)
```

## ライブリージョン：動的なUIの通知

UIが動的に変化するとき、スクリーンリーダーにその変化を伝える必要がある。例えば、フォームのバリデーションエラーが表示されたとき、スクリーンリーダーユーザーは画面を見ていないためそのエラーに気づかない可能性がある。

Flutterでは `SemanticsService.announce()` を使って、スクリーンリーダーに対してアナウンスを送ることができる。

```dart
// フォームエラー表示時にアナウンスする
void _showError(String message) {
  setState(() => _errorMessage = message);
  SemanticsService.announce(message, TextDirection.ltr);
}
```

また `liveRegion` プロパティを `Semantics` ウィジェットに設定することで、その要素の内容が変わったときに自動的にアナウンスされるようになる。

## アクセシビリティのテスト

Golden Test（ビジュアルスナップショット）がUIの見た目を守るように、アクセシビリティに対しても自動テストを組み込むことができる。

Flutter には `tester.getSemantics()` を使ってセマンティクスノードを取得し、その内容をアサートするAPIがある。

```dart
testWidgets('お気に入りボタンに正しいラベルが付いている', (tester) async {
  await tester.pumpWidget(FavoriteButton(isFavorited: false));

  final semantics = tester.getSemantics(find.byType(FavoriteButton));
  expect(semantics.label, 'お気に入りに追加');
  expect(semantics.hasFlag(SemanticsFlag.isButton), isTrue);
});
```

このようなテストを書くことで、UIのリファクタリング時にセマンティクスが壊れていないことを継続的に確認できる。

## まとめ

FlutterのSemanticsは、スクリーンリーダーのための「付け足し」ではなく、UIの意味構造を定義するものだ。丁寧に設計されたセマンティクスツリーは、アクセシビリティを必要とするユーザーを助けるだけでなく、UIが正しく意味を持っているかというQAの観点からも価値がある。

- 標準ウィジェットのデフォルトセマンティクスを信頼しつつ、カスタムUIには明示的に`Semantics`を設定する
- `ExcludeSemantics`で装飾的な要素をツリーから除外する
- フォーカス順序を論理的に設計する
- 動的なUIの変化は`SemanticsService.announce()`で通知する
- テストでセマンティクスのリグレッションを防ぐ

アクセシビリティは後から追加するものではなく、UIを設計する段階から考えるべきものだ。Semanticsの理解を深めることで、より多くの人に使ってもらえるアプリを作れるようになる。
