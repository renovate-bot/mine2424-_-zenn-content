# Flutterのウィジェットライフサイクルを理解してパフォーマンスを改善する

## はじめに

Flutterのパフォーマンス問題に直面したとき、多くの開発者がまず「不要なrebuildを減らす」というアドバイスを目にする。Riverpodの `select`、`const` コンストラクタ、`RepaintBoundary`——様々な手法が紹介される。

しかし「なぜ不要なrebuildが起きるのか」「そもそもrebuildとは何か」を理解していないと、手法を試してみても効果が出なかったり、新たな問題を生んだりすることがある。

この記事では、Flutterのウィジェットライフサイクルの仕組みを丁寧に追いながら、パフォーマンス改善の考え方を整理する。

---

## Flutterのウィジェットはどう動くか

Flutterの描画システムを理解する上で、まず三つの概念を押さえる必要がある。

**Widget**
UIを記述した設定書のようなもの。`build` メソッドが返す軽量なオブジェクト。イミュータブルで、頻繁に作り直される。

**Element**
ウィジェットツリーの実体。ウィジェットとRenderObjectを繋ぐ存在。状態（Stateオブジェクト）を保持する。

**RenderObject**
実際のレイアウト計算と描画を担う。コストが高い処理をする。

この三層構造が重要なのは、「rebuild（ウィジェットの再生成）」と「再描画（RenderObjectの更新）」が別の操作だからだ。

---

## rebuildは「再描画」ではない

Flutterのパフォーマンス話でよく誤解されるのが、「rebuildが起きると重い」という認識だ。

`setState` を呼ぶと `build` メソッドが再実行され、新しいウィジェットオブジェクトが生成される。これが「rebuild」だ。しかしウィジェットオブジェクトはDartのオブジェクトとして軽量で、生成自体のコストは比較的小さい。

本当にコストが高いのは、RenderObjectの更新だ。Flutterは新旧のウィジェットツリーを比較し（diffing）、実際に変化があった部分だけRenderObjectを更新する。この比較処理と、変化があった部分の再レイアウト・再描画が、実際のパフォーマンスコストの大部分を占める。

つまり、「rebuildを減らす」ことよりも「RenderObjectの更新を最小化する」ことの方が本質的だ。

---

## StatefulWidgetのライフサイクル

`StatefulWidget` には明確なライフサイクルがある。各フェーズを理解することで、適切な処理を適切なタイミングに配置できる。

```dart
class MyWidget extends StatefulWidget {
  const MyWidget({super.key});
  
  @override
  State<MyWidget> createState() => _MyWidgetState();
}

class _MyWidgetState extends State<MyWidget> {
  
  @override
  void initState() {
    super.initState();
    // ウィジェットがツリーに挿入された直後
    // API呼び出しの開始、StreamのSubscribe、AnimationControllerの初期化など
  }
  
  @override
  void didChangeDependencies() {
    super.didChangeDependencies();
    // 依存するInheritedWidgetが変化したとき
    // ここでBuildContextに依存した処理を行う（initStateではContextは安全でない）
  }
  
  @override
  void didUpdateWidget(MyWidget oldWidget) {
    super.didUpdateWidget(oldWidget);
    // 親からの新しいウィジェットで更新されたとき
    // oldWidgetと比較して、必要な場合だけ処理を行う
  }
  
  @override
  Widget build(BuildContext context) {
    return const SizedBox();
  }
  
  @override
  void dispose() {
    // ウィジェットがツリーから削除されたとき
    // Controllerの破棄、Subscriptionのキャンセルなど
    super.dispose();
  }
}
```

**パフォーマンスへの影響が大きいのは `didUpdateWidget`だ。**

親ウィジェットがrebuildされると、子ウィジェットも再生成される。そのとき `didUpdateWidget` が呼ばれる。もしここで `setState` を呼ぶと連鎖的なrebuildが起きる。`oldWidget` と `widget` を比較して、本当に必要な場合だけ処理する習慣が重要だ。

---

## 「const」が効く理由

`const` コンストラクタを使うとパフォーマンスが改善されると言われる。その理由は、Flutterのdiffingアルゴリズムにある。

Flutterは新旧のウィジェットを比較するとき、まず「同一インスタンスか」をチェックする。`const` で作られたウィジェットはコンパイル時に定数として確定されるため、同じ `const` 式からは同一インスタンスが返される。

つまりFlutterは「このウィジェットは前回と同じインスタンスだ→変化なし」と即座に判断でき、それ以下のサブツリー全体のdiffingをスキップできる。

これが `const` の本質だ。「軽くなる」のではなく「比較をスキップできる」ことが効く。

---

## どこでパフォーマンスが落ちるか見極める

感覚でパフォーマンス改善を進めるのは効率が悪い。Flutter DevToolsの「Performance」タブを使うと、どのフレームで何がrebuildされているかを可視化できる。

特に「Widget Rebuild Stats」は有用だ。頻繁にrebuildされているウィジェットを特定し、なぜ頻繁にrebuildされるのかを追跡する。

よくあるパターンとして：

- **ListViewの中に重いウィジェット**：スクロールのたびに全アイテムがrebuildされる。`ListView.builder` を使い、見えている範囲だけを構築する。

- **アニメーション中の大規模rebuild**：`AnimatedBuilder` を使い、アニメーションの変化を受け取るウィジェットの範囲を限定する。

- **状態管理の粒度が粗い**：グローバルな状態の一部が変化したとき、関係のない多くのウィジェットもrebuildされる。状態を細かく分割し、必要な部分だけを購読させる。

---

## 「過剰最適化」に気をつける

パフォーマンス改善の議論で見落とされがちなのが、「最適化自体のコスト」だ。

`const` を使えるところには使う、状態を細かく分割する——これらは確かに有効だが、あらゆる場所に適用しようとするとコードが複雑になり、可読性が下がり、保守コストが上がる。

Flutterは60fps（多くのデバイスで120fps）を維持するために1フレームあたり16ms（8ms）以内に処理を収める必要がある。まずその基準を満たしているかを測定で確認し、実際にフレームドロップが起きている箇所を特定してから最適化する。

体感でも測定でも問題がない部分を「最適化できそうだから」と変更するのは、時間と複雑性のコストを無駄に払うことになる。

---

## まとめ

Flutterのウィジェットライフサイクルを理解することで、パフォーマンス問題の根本原因を特定しやすくなる。

重要なのは「rebuildを減らす」ことよりも「RenderObjectの更新を最小化すること」であり、そのために `const` によるdiffingスキップ、ライフサイクルの適切な活用、状態購読の粒度の調整がある。

そして最適化は、測定によって問題を特定してから行う——この原則はFlutterに限らず、あらゆる最適化の基本だ。
