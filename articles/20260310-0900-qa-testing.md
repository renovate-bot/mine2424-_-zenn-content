## はじめに

フロントエンドのテストで「APIのモック」をどう扱うかは、長年悩ましい問題だ。`jest.mock()` でaxiosを差し替えたり、fetchをスパイしたり——これらは動くが、テストコードがモック実装で汚染され、「本物のAPIと同じ振る舞いをしているか」という確信が持てない。

MSW（Mock Service Worker）はその問題を根本から解決するアプローチを提示する。「ネットワークレベルでリクエストを横取りする」という発想だ。

## MSWが解決する本質的な問題

従来のAPIモックには2つの根本的な問題がある。

**実装の詳細に依存している**。`axios.get` を `jest.fn()` で差し替えるということは、「axiosを使っている」という実装の詳細がテストコードに漏れている。将来的にfetchやSWRに変更したとき、テストも書き直しが必要になる。

**モックが本物と乖離しやすい**。モックは手動で管理する。バックエンドのAPIが変わったとき、テストのモックが古いままになることは珍しくない。テストは通るのに本番は壊れている、という状況が起きる。

MSWはどちらも解決する。Serviceworker（ブラウザ）やNode.jsのHTTPインターセプター（テスト環境）のレベルでリクエストを捕まえるため、**アプリケーションコードは本物のAPIを呼ぶように書いたまま**でいい。axiosだろうがfetchだろうが、内部実装に関係なくモックが効く。

## ハンドラーが「仕様書」になる

MSWの中心はハンドラーの定義だ。

```typescript
import { http, HttpResponse } from 'msw'

export const handlers = [
  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json({
      id: params.id,
      name: 'テストユーザー',
      email: 'test@example.com'
    })
  }),
  
  http.post('/api/users', async ({ request }) => {
    const body = await request.json()
    return HttpResponse.json({ id: '123', ...body }, { status: 201 })
  })
]
```

このハンドラーはURL・メソッド・レスポンス形式を明示的に定義する。これは単なるモックコードではなく、**APIの仕様を表現したもの**として機能する。OpenAPI仕様に近い役割を持つと考えるといい。

バックエンドのレスポンス形式が変わったとき、ここを更新すればすべてのテストに反映される。散在した `jest.fn()` を探し回る必要がない。

## テスト環境とブラウザ環境の統一

MSWの優れた点は、**同じハンドラーをテスト環境と開発環境の両方で使える**ことだ。

テスト環境（Node.js）では `setupServer` を使う。

```typescript
import { setupServer } from 'msw/node'
import { handlers } from './handlers'

const server = setupServer(...handlers)

beforeAll(() => server.listen())
afterEach(() => server.resetHandlers())
afterAll(() => server.close())
```

ブラウザの開発環境（バックエンドが未実装の状態）では `setupWorker` を使う。同じhandlers.tsを参照するだけで、開発中もAPIなしで動くモックサーバーが手に入る。

「テストは通るけど開発サーバーでは動かない」という乖離がなくなる。テスト・開発・本番の三者が同じ契約の上に立てる。

## エラーケースのテストが書きやすくなる

MSWのもう一つの強みはエラーシナリオのテストだ。「サーバーが500を返したとき」「ネットワークエラーが起きたとき」——これらをテストすることは重要だが、従来のモックでは煩雑になりがちだった。

MSWではテストごとにハンドラーを上書きできる。

```typescript
it('APIエラー時にエラーメッセージを表示する', async () => {
  server.use(
    http.get('/api/users/:id', () => {
      return HttpResponse.json(
        { message: 'Internal Server Error' },
        { status: 500 }
      )
    })
  )
  
  // テスト実行
})
```

`server.use()` でハンドラーを一時的に上書きし、テスト後は `resetHandlers()` で元に戻る。エラーケースのテストが宣言的に書けるようになる。

## MSWとコントラクトテストの関係

MSWはモックツールだが、適切に使うとコントラクトテストに近い効果が得られる。

ポイントはハンドラーの定義をOpenAPI仕様やバックエンドのスキーマと同期させることだ。`openapi-msw` のようなツールを使えば、OpenAPIの定義からMSWハンドラーを自動生成できる。バックエンドとの乖離を自動的に検出できる仕組みに発展する。

これは「モックがドキュメントになる」という逆転の発想だ。

## いつMSWを使うべきか

MSWが特に有効なのは次のシナリオだ。

**フロントエンドとバックエンドが並行開発されているとき**。API仕様が決まった段階でハンドラーを定義すれば、バックエンドの完成を待たずに開発・テストが進められる。

**多くのコンポーネントが同じAPIに依存しているとき**。モックの定義が一箇所に集約されるため、変更コストが下がる。

**統合テスト（Integration Test）を書きたいとき**。ユニットテストでは細かく差し替えるより、MSWで「APIとのインタラクション全体」をテストしたほうが信頼性が高いケースがある。

逆に、純粋なユニットテストで特定の関数の振る舞いだけを確認したい場合は、従来の `jest.fn()` のほうがシンプルなこともある。

## まとめ

MSWの本質は「テストコードを実装の詳細から切り離す」ことだ。HTTPレベルでリクエストを捕まえることで、アプリケーションのコードを変えることなくモックが効く。ハンドラーはAPIの仕様書として機能し、テストと開発環境を統一できる。

「APIのモックをどこに書くか」という問いの答えは、MSWによって「ネットワーク境界に書く」という形に収束しつつある。フロントエンドのテスト戦略を見直すなら、まずMSWの思想から理解してみてほしい。
