## 「URLが変わらないUI」の難しさ

モダンなWebアプリには、URLが変わりながらも「背後のコンテンツを保持する」UIが求められるケースがある。

たとえばTwitter（X）の画像拡大表示。フィードを眺めているとき、画像をクリックすると**モーダルが開いてURLが変わる**が、背景にはフィードが透けて見えている。そしてモーダルを閉じると元のフィードに戻る。

しかし、そのURLを直接開くと、モーダルではなく**画像単独の専用ページ**が表示される。

これを普通のルーティングで実装しようとすると、複雑な状態管理やURLのハック的な操作が必要になる。Next.js App Routerの**Parallel Routes**と**Intercepting Routes**は、この種のUIをファイルシステムのルール上でエレガントに実現するための仕組みだ。

---

## Parallel Routes（並行ルート）とは

Parallel Routesは、**同じレイアウト内に複数のページコンポーネントを並行して表示する**仕組みだ。

`@`プレフィックスを持つフォルダが「スロット」になる。

```
app/
  layout.tsx        ← ルートレイアウト
  @sidebar/         ← サイドバースロット
    page.tsx
  @main/            ← メインコンテンツスロット
    page.tsx
  page.tsx          ← デフォルトページ
```

`layout.tsx` はスロットをpropsとして受け取る。

```tsx
export default function Layout({
  children,
  sidebar,
  main,
}: {
  children: React.ReactNode;
  sidebar: React.ReactNode;
  main: React.ReactNode;
}) {
  return (
    <div className="flex">
      <aside>{sidebar}</aside>
      <main>{main}</main>
    </div>
  );
}
```

ポイントは、**各スロットが独立したナビゲーション状態を持てること**だ。サイドバーとメインコンテンツが独立してURLに対応し、片方が変わっても他方は維持される。

### どんなときに使うか

- ダッシュボード：左サイドバーのナビゲーションとメインコンテンツが独立して動く
- 比較UI：2つのドキュメントやビューを並べて表示する
- タブ付きインターフェース：複数の並行するビューをURL対応させたい
- 条件分岐によるUI：認証状態によってレイアウト内の表示を切り替える

### `default.tsx` の役割

Parallel Routesでは、**サブナビゲーション時にスロットが「どこにいるかわからない」状態**が生じうる。たとえばメインコンテンツのルートが変わったとき、サイドバースロットは「現在のパスに対応するページを持っていない」ケースがある。

こういう場合に `default.tsx` がフォールバックとして機能する。スロットに `default.tsx` を用意しておくことで、対応するルートがない場合の表示を定義できる。

## Intercepting Routes（インターセプトルート）とは

Intercepting Routesは、**あるURLへのナビゲーションを「内部から」インターセプトし、別のコンポーネントを代わりに表示する**仕組みだ。ただし、外部から直接アクセスされた場合（URLをブラウザに直接入力した場合）は、本来のコンポーネントが表示される。

フォルダ名に`(.)`などのプレフィックスを付けることでインターセプトを定義する。

```
(.) — 同じレベルのルートをインターセプト
(..) — 1つ上のレベルのルートをインターセプト
(..)(..) — 2つ上のレベルをインターセプト
(...) — appルートからインターセプト
```

```
app/
  feed/
    page.tsx          ← フィードページ
  photo/
    [id]/
      page.tsx        ← 写真の単独ページ（直接アクセス時）
  @modal/
    (.)photo/
      [id]/
        page.tsx      ← フィードからアクセス時はモーダルとして表示
    default.tsx       ← モーダルなし状態
```

### 動作のイメージ

1. ユーザーが `/feed` を閲覧中
2. 写真リンクをクリック → URLが `/photo/123` に変わる
3. **Intercepting Routeが発動** → `@modal/(.)photo/[id]/page.tsx` が表示される（モーダルUI）
4. 背景には `/feed` のコンテンツが残る
5. ページをリロード or URL直接アクセス → `photo/[id]/page.tsx` が表示される（専用ページ）

これがTwitterの画像拡大UIと同じパターンだ。

## Parallel Routes × Intercepting Routesの組み合わせ

2つの機能は組み合わせて使うことで真価を発揮する。インターセプトしたルートを**モーダルスロット**に表示することで、背景コンテンツを保ちながらURL対応モーダルを実現できる。

```
app/
  layout.tsx          ← @modalを受け取るレイアウト
  @modal/
    (.)photo/
      [id]/
        page.tsx      ← モーダルの中身
    default.tsx       ← null（モーダルなし）
  feed/
    page.tsx
  photo/
    [id]/
      page.tsx        ← 直接アクセス時の専用ページ
```

`layout.tsx`：

```tsx
export default function RootLayout({
  children,
  modal,
}: {
  children: React.ReactNode;
  modal: React.ReactNode;
}) {
  return (
    <html>
      <body>
        {children}
        {modal}  {/* モーダルがある場合だけ表示される */}
      </body>
    </html>
  );
}
```

`@modal/default.tsx`：

```tsx
export default function Default() {
  return null;  // モーダルが不要な場合は何も表示しない
}
```

`@modal/(.)photo/[id]/page.tsx`（モーダルUI）：

```tsx
'use client';
import { useRouter } from 'next/navigation';
import Modal from '@/components/Modal';
import PhotoView from '@/components/PhotoView';

export default function PhotoModal({ params }: { params: { id: string } }) {
  const router = useRouter();

  return (
    <Modal onClose={() => router.back()}>
      <PhotoView id={params.id} />
    </Modal>
  );
}
```

## この仕組みが解決すること

### 共有可能なURL

モーダル状態がURLに反映されるため、**URLをシェアするとモーダルの内容に直接アクセスできる**。受け取った側は専用ページとして表示される。ソーシャルシェアとの相性が非常に良い。

### ブラウザの戻るボタン対応

`router.back()` でモーダルを閉じれば、ブラウザの「戻る」ボタンで自然にフィードに戻る。SPAらしい自然なナビゲーション体験をURLベースで実現できる。

### SEO・アクセシビリティとの両立

専用ページとして直接アクセス可能なため、クローラーに対してもコンテンツが公開できる。サーバーコンポーネントとして実装すれば、OGPなどのメタデータも専用ページ側で適切に設定できる。

---

## 設計上の注意点

**複雑さとのトレードオフ**：Parallel RoutesとIntercepting Routesはファイル構造が複雑になる。小さなアプリに導入すると、得られるメリットよりも理解コストの方が大きくなりうる。本当にURLベースのモーダルや並行ビューが必要なケースに絞って使う判断が重要だ。

**`default.tsx` の管理**：各スロットに適切な `default.tsx` がないと、ナビゲーション時に予期しないエラーになる。ファイル構造が増えた際に `default.tsx` を忘れがちなので注意が必要だ。

**loading・error の設計**：各スロットは独立して `loading.tsx` と `error.tsx` を持てる。並行表示されているコンポーネントのローディング状態やエラーを適切にハンドリングするため、スロットごとにこれらを設計する意識が必要だ。

---

## まとめ

Next.js App RouterのParallel RoutesとIntercepting Routesは、「モーダルをURLに対応させたい」「並行するビューをファイルシステムで管理したい」という要求に対して、フレームワーク側が設計的な解答を示したものだ。

使い所を見極めることが大切だが、一度理解すると「こういうUIはファイル構造で表現できるんだ」という発見がある。Next.jsのルーティングがただのURLマッピングではなく、**UIの構造そのもの**を表現するための言語になっていることを実感できる機能だ。
