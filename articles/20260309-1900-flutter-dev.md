## はじめに：Flutter Webへの誤解

Flutterはクロスプラットフォームフレームワークだ。iOS、Android、Web、Desktop——一つのコードベースから複数のプラットフォームに向けてアプリを届けられる。この特性は魅力的だが、「同じコードがどのプラットフォームでも最高に動く」という意味ではない。

Flutter Webはこの誤解を最も受けやすい領域だ。FlutterモバイルアプリをそのままWebに展開しようとして、パフォーマンスやSEO、アクセシビリティの問題に直面するチームは多い。

Flutter WebをWebとして正しく設計するには、モバイルアプリとWebの本質的な違いを理解し、Flutter Webの特性——制約と強みの両方——を把握することが出発点になる。

## Flutter WebのレンダリングエンジンとHTML/CanvasKit

Flutter Webには2つのレンダリングエンジンがある：HTMLレンダラーとCanvasKitだ。

**HTMLレンダラー**はFlutterのウィジェットをブラウザのHTML要素とCSS、Canvasを組み合わせてレンダリングする。バンドルサイズが小さく初期ロードが速い。SEOの観点からもHTML要素が生成されるため、クローラーがコンテンツを読みやすい。

**CanvasKit**はSkia（Flutterモバイルが使うグラフィックエンジン）をWebAssemblyにコンパイルして使う。ピクセル単位で一致するレンダリングと高度なグラフィック表現が可能だが、初期ロード時にWebAssemblyバイナリ（約1.5MB以上）をダウンロードする必要がある。

選択の基準は用途によって変わる。コンテンツ中心のWebアプリや一般向けのサービスにはHTMLレンダラー、グラフィック集中的なツールや社内ダッシュボードにはCanvasKitというのが基本的な判断軸だ。

Flutter 3以降では、自動選択（Auto）がデフォルトになっており、モバイルブラウザではHTMLレンダラー、デスクトップブラウザではCanvasKitを使う。ただし、プロダクトの特性を理解した上で明示的に選択する方が意図した結果が得られる。

## 初期ロードの設計

WebアプリでFlutterが最も批判を受けやすいのが初期ロードのパフォーマンスだ。

Flutterのビルド成果物（main.dart.js）は最小化しても数MB規模になることが多い。これはネイティブのWebアプリと比較すると重い。対策として考えられる方向性はいくつかある。

**deferred imports（遅延インポート）**を活用することで、初期ロードに不要なコードを分割して後からロードできる。Dartの `deferred as` 構文を使うと、特定ルートや機能のコードを必要になるまでダウンロードしない設計が可能だ。

```dart
import 'package:my_app/heavy_feature.dart' deferred as heavyFeature;

Future<void> loadFeature() async {
  await heavyFeature.loadLibrary();
  heavyFeature.initialize();
}
```

**Service Workerによるキャッシュ**も重要だ。FlutterのWebビルドはデフォルトでService Workerを生成するが、キャッシュ戦略を理解してカスタマイズすることで、初回ロード後のリピートアクセスを大幅に高速化できる。

また、**ローディング体験のカスタマイズ**も見落とされがちだ。`flutter.js` のローディングフックを使うと、Flutterエンジンの初期化中に表示するスプラッシュ画面やプログレスインジケーターをカスタマイズできる。ユーザーに「読み込み中」を意識させない体験設計が、体感的なパフォーマンスに大きく影響する。

## SEOとクローラビリティ

Flutter Webの重大な制約の一つがSEOだ。

CanvasKitはcanvas要素に全てを描画するため、クローラーがテキストコンテンツを読めない。SEOが重要なページにCanvasKitは向いていない。HTMLレンダラーを使えばある程度のHTML構造が生成されるが、それでもネイティブHTMLと比べてクローラーが解析しやすいとは言えない。

この制約に対する現実的な対応は2つある。

一つは、SEOが重要なページ（ランディングページ、記事ページ）は純粋なHTML/CSS/JSやNext.jsで実装し、FlutterはSEO要件のない部分（認証後のダッシュボード、インタラクティブなツール）に限定するというアーキテクチャの分離だ。

もう一つは、meta tagやOpen Graphのような基本的なSEO要素を`index.html`に直接埋め込み、クローラーが読む情報を最低限確保するというアプローチだ。ダイナミックなコンテンツのSEOには対応できないが、静的な情報であれば機能する。

## URLとルーティングの設計

WebアプリとしてのFlutter Webでは、URLの設計が重要になる。

go_routerはFlutterの標準的なルーティングライブラリだが、Web特有の考慮が必要だ。URLが変わらないSPA的な動作では、ブラウザの戻る/進むボタンが正しく機能しない。go_routerのHistoryモードを適切に設定し、各画面が固有のURLを持つよう設計することが基本だ。

また、初回アクセスが特定のURLに来たとき（直リンクやリロード）にも同じページが表示されるよう、Webサーバーの設定とFlutter側のルーティングが整合している必要がある。静的ホスティング（Firebase Hosting、Cloudflare Pages）では、全てのパスをindex.htmlにリダイレクトするルールの設定が必要だ。

## アクセシビリティ——WebはMobileと異なる

Flutterの `Semantics` ウィジェットはモバイルのスクリーンリーダーのためだけでなく、Web上のアクセシビリティ（ARIA属性）にも影響する。

HTMLレンダラーを使用した場合、Semanticsの情報はHTML要素のARIA属性として出力される。スクリーンリーダーがナビゲートできる構造を持つためには、モバイルと同様にSemanticsを丁寧に設定する必要がある。

`ExcludeSemantics` と `MergeSemantics` を使いながら、どの情報をアクセシビリティツリーに含めるかを意識的に設計することが、Webアクセシビリティの観点では特に重要だ。

## Flutter Webを選ぶ判断基準

実際のプロジェクトでFlutter Webを選ぶべきかどうかは、慎重に判断したい。

向いている用途は、複雑なインタラクティブアプリ（データ可視化ツール、設計ツール、ゲーム）、社内向けダッシュボードやB2B SaaS、モバイルとWebで全く同じ体験を提供する必要があるケース、などだ。

一方、SEOが重要なコンテンツサイト、軽量なマーケティングページ、外部APIを多用するシンプルなWebアプリ——こういった用途ではFlutter Webの恩恵より制約の方が目立つ可能性が高い。

## おわりに

Flutter Webは「モバイルアプリがそのままWebになる」ではなく「Webとして設計した時に強みが出る」技術だ。

レンダリングエンジンの選択、初期ロードの最適化、SEOの制約理解、URLとルーティング設計、アクセシビリティ——これらの設計判断は全てWebの文脈で考える必要がある。

Flutterの「一つのコードベースで複数プラットフォーム」というビジョンは魅力的だが、プラットフォームごとの文化と制約を無視しては良い体験は作れない。Flutter Webを選んだなら、Webを深く理解することが、良いFlutter Webアプリを作る前提条件だ。
