## はじめに

Reactを経験してからFlutterを学んだ開発者がまず感じるのは、「StatefulWidgetの冗長さ」だろう。`initState`、`dispose`、`setState`——状態とライフサイクルのコードがウィジェットクラスの至る所に散らばる。ReactにはHooksという解決策があった。Flutterにもある。それが `flutter_hooks` だ。

このパッケージは単なるReactの模倣ではない。Flutterのウィジェットツリーという独自の世界で、「状態とロジックをどこに書くか」という問いに新しい答えを提示している。

## StatefulWidgetの「儀式」を思い出す

通常のFlutterでアニメーションを実装するとき、こんなコードを書く：

```dart
class MyWidget extends StatefulWidget {
  const MyWidget({super.key});
  @override
  State<MyWidget> createState() => _MyWidgetState();
}

class _MyWidgetState extends State<MyWidget>
    with SingleTickerProviderStateMixin {
  late AnimationController _controller;

  @override
  void initState() {
    super.initState();
    _controller = AnimationController(
      vsync: this,
      duration: const Duration(milliseconds: 300),
    );
  }

  @override
  void dispose() {
    _controller.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    // ようやくここでUIを書く
    return AnimatedBuilder(
      animation: _controller,
      builder: (context, child) => /* ... */,
    );
  }
}
```

これはFlutterが正式にサポートするパターンであり、間違いではない。ただ、書くべきことが多い。`initState` と `dispose` は常にペアで現れ、どちらかを忘れるとリソースリークが生まれる。コードを読む側も、`initState` → `dispose` → `build` と複数箇所を行き来しながら全体像を把握する必要がある。

## flutter_hooksが変えること

`flutter_hooks` を使うと、同じロジックをこう書ける：

```dart
class MyWidget extends HookWidget {
  const MyWidget({super.key});

  @override
  Widget build(BuildContext context) {
    final controller = useAnimationController(
      duration: const Duration(milliseconds: 300),
    );
    // controllerはウィジェットの生存期間と同期して自動でdisposeされる
    return AnimatedBuilder(
      animation: controller,
      builder: (context, child) => /* ... */,
    );
  }
}
```

`initState` も `dispose` もない。`useAnimationController` が呼ばれた瞬間にコントローラが作られ、ウィジェットがツリーから外れたときに自動で破棄される。生成と破棄が「1箇所のコード」に凝縮されている。

## Hooksの仕組み——なぜこれで動くのか

Hooksが「魔法」に見える理由の一つは、どうやってウィジェットのライフサイクルと同期しているのかが見えにくいからだ。実装を理解すると、設計の意図が見えてくる。

`HookWidget` は `build` を直接実装せず、内部でHooksの状態を管理する仕組みを持つ。`use*` 系の関数が呼ばれるたびに、ウィジェットに紐づいたリストに状態オブジェクトが登録されていく。このリストは `build` が呼ばれるたびに同じ順序で参照される。

**重要な制約**：Hooksは常に同じ順序で呼ばれなければならない。条件分岐やループの中でHooksを呼ぶことはできない。これはReact Hooksと同じルールだ。

```dart
// NG: 条件の中でHookを呼ぶ
if (someCondition) {
  final value = useState(0); // ルール違反
}

// OK: 常に同じ順序で
final value = useState(0);
if (someCondition) {
  value.value = 1; // 値を変えるのはOK
}
```

このルールがある理由は、Hooksがインデックスベースで状態を管理しているからだ。呼ばれる順序が変わると、どのインデックスがどの状態かがずれてしまう。

## よく使うHooks一覧

`flutter_hooks` が提供するHooksは豊富だ。実務でよく使うものを整理しておく。

**`useState`** はもっとも基本的なHook。`setState` を直接扱う代わりに、変更可能な値オブジェクトを返す。

**`useEffect`** は副作用を扱う。ウィジェットの初回構築時や特定の値が変わったタイミングで処理を実行し、クリーンアップ関数を返すこともできる。Flutterの `initState` + `didUpdateWidget` + `dispose` の組み合わせを1つで表現できる。

**`useMemoized`** は高コストな計算結果をキャッシュする。依存値が変わるまで再計算しない。

**`useRef`** は再レンダリングを引き起こさずに値を保持する。前回の値を記憶したいときなどに使う。

**`useTextEditingController`** や **`useScrollController`** など、よく使うFlutterコントローラ用のHooksも揃っている。これらは全て自動でdisposeされる。

## カスタムHooksで「ロジックを抽出する」

Hooksの真価は、カスタムHooksを作れることにある。「ある状態とロジックのまとまり」を関数として切り出し、複数のウィジェットから再利用できる。

例えば、ネットワークリクエストの状態管理（ローディング・成功・エラー）を扱うカスタムHookを作れば、各画面でボイラープレートを繰り返す必要がなくなる。命名規則として `use` プレフィックスをつけることで、呼び出し側は「これはHookだ」と一目でわかる。

従来のStatefulWidgetでは、こうしたロジックの再利用は `mixin` を使うか、ロジックを外部クラスに切り出してウィジェットから参照するかのどちらかだった。カスタムHooksはより直感的な形で同じゴールを達成する。

## Riverpodとの使い分け

`flutter_hooks` は Riverpod（`hooks_riverpod` パッケージ）と組み合わせることが多い。この2つは役割が異なる。

- **flutter_hooks**: ウィジェットローカルな状態とライフサイクルを扱う
- **Riverpod**: アプリ全体で共有する状態や非同期データを扱う

アニメーションコントローラ、テキストフィールドの入力値、フォームの一時的な状態——これらはウィジェットが消えれば不要になるローカルな状態だ。`flutter_hooks` が輝くのはここだ。

一方、ユーザーのログイン状態、取得したAPIデータ、複数画面で共有するカートの内容——これらはウィジェットのライフサイクルとは切り離して管理したい。Riverpodの領域だ。

`hooks_riverpod` を使えば `HookConsumerWidget` として両方を組み合わせられる。`useProvider`（現在は `ref.watch`）でRiverpodの状態を購読しながら、Hooksでローカルなアニメーションも管理する、という構成が自然に書ける。

## 導入を検討すべきタイミング

すべてのプロジェクトに `flutter_hooks` が必要なわけではない。導入を検討する判断軸をいくつか挙げる。

**向いているプロジェクト**：
- ReactやVue Compositionに慣れたメンバーが多いチーム
- アニメーションを多用するUIで、コントローラの管理が煩雑になっている
- ウィジェットをシンプルな関数型スタイルで書きたい

**慎重になるべきケース**：
- Flutterに不慣れなメンバーが多い。Hooksのルール（順序を変えてはいけない）を理解していないとデバッグが難しくなる
- チームで「StatefulWidgetに統一する」という合意がある
- パッケージの依存を増やしたくない

使いたい理由が「コードが短くなる」だけなら、一度立ち止まるといい。`flutter_hooks` は可読性とシンプルさを高めるツールだが、チームに馴染みがなければ逆に読みにくくなる。

## おわりに

`flutter_hooks` が提供するのは、「状態とそのライフサイクルを近くに書く」という設計思想だ。`initState` と `dispose` が離れた場所に存在する従来の書き方は、Flutterが正しいと認めたパターンではある。ただ、関連するコードは近くにあるべきという原則から見れば、Hooksのアプローチには説得力がある。

Reactから来た開発者なら、この考え方はすぐに腑に落ちるだろう。Flutter専任だった開発者にとっては、最初は違和感があるかもしれない。どちらにせよ、「状態をどう管理するか」という問いに対する一つの良い答えとして、知っておく価値がある選択肢だ。
