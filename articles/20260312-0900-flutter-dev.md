## メモリリークは「静かなバグ」だ

クラッシュするバグは目に見える。テストが落ちるバグも検出できる。しかしメモリリークは違う。アプリは動き続け、UIも正常に見える。ただ、時間が経つにつれてメモリ使用量が増え、やがてOSに強制終了される。ユーザーは「なんかたまに落ちる」と感じるだけで、原因を突き止めるのは難しい。

Flutterアプリにおけるメモリリークの主な原因は「解放されないオブジェクトの参照」だ。Dartのガベージコレクター（GC）はリフレンスが残っているオブジェクトを解放できない。Streamのサブスクリプションをキャンセルしないまま`StatefulWidget`が`dispose()`されたり、`AnimationController`が破棄されなかったりすると、ウィジェットは画面から消えてもオブジェクトはメモリに居座り続ける。

この記事では、Flutterのメモリリークをどう検出し、どう設計すれば防げるのかという思想を整理する。

## DevTools Memory Profileを「読む」練習

Flutter DevToolsのMemoryタブは、アプリのヒープ（ヒープメモリ）の状態をリアルタイムで可視化する。数字を眺めるより、「パターン」を読むことが大切だ。

正常なアプリのメモリグラフは「鋸歯状」になる。オブジェクトが生成されてメモリが増え、GCが走ると急に下がる。このサイクルが繰り返される。重要なのは、**GCの後もメモリが元の水準まで下がるかどうか**だ。

リークが起きているアプリでは、GCを経ても「床」が少しずつ上がっていく。特定の操作（画面遷移、リストスクロールなど）を繰り返すたびに最低ラインが上昇するなら、その操作でオブジェクトが積み上がっているサインだ。

DevToolsにはスナップショット機能もある。2つのタイミングでスナップショットを撮り、差分を見ることで「増えたオブジェクト」の種類と数を特定できる。ここで`MyWidget`や`StreamSubscription`のインスタンス数が操作ごとに増えているなら、そのクラスがリークの原因候補になる。

## 典型的なリークパターンと設計レベルの対策

### 1. StreamSubscriptionの未解除

非同期データを扱うとき、`StreamController`や`BehaviorSubject`のサブスクリプションは手動でキャンセルする必要がある。

```dart
class _MyWidgetState extends State<MyWidget> {
  late StreamSubscription _subscription;

  @override
  void initState() {
    super.initState();
    _subscription = someStream.listen((event) {
      // ...
    });
  }

  @override
  void dispose() {
    _subscription.cancel(); // これを忘れるとリーク
    super.dispose();
  }
}
```

この問題への設計レベルの対策は「Streamを`StatefulWidget`で直接購読しない」ことだ。`StreamBuilder`ウィジェットを使えば、ウィジェットの生存期間に合わせてサブスクリプションが自動管理される。RiverpodやBlocなどの状態管理ライブラリも、この責務をフレームワーク側に引き受ける設計になっている。

### 2. AnimationControllerの未dispose

`AnimationController`は`TickerProvider`と結びついており、破棄しないとティッカーが動き続ける。`SingleTickerProviderStateMixin`を使う場合は必ず`dispose()`で解放する必要がある。

AnimationControllerを多用するコンポーネントでは、`dispose()`の実装漏れが起きやすい。「アニメーションを持つウィジェットには必ずDisposeチェックをCI/CDに組み込む」という設計規約を設けることが、チーム開発では有効だ。

### 3. グローバルなコールバック参照

`GlobalKey`や`singleton`パターンで保持されたオブジェクトが、ウィジェットへのコールバックを持っている場合、ウィジェットが破棄されてもコールバックを通じて参照が生き続ける。

設計としては「コールバックを長寿命オブジェクトに渡さない」か、`WeakReference`（Dart 2.17以降）を使うかを検討する。ただし`WeakReference`は参照が消えていることを前提にしたコードを書く必要があり、設計が複雑になる。むしろアーキテクチャの見直し（シングルトンへの依存を減らす、DIコンテナで生存期間を管理する）の方が根本的な解決になることが多い。

## leak_trackerで自動検出する

Flutter SDK（Flutter 3.18以降）には`leak_tracker`という仕組みが組み込まれている。これはテスト中にWidgetやDisposableオブジェクトが適切に`dispose()`されているかを自動チェックする。

テストで`testWidgets`を使うとき、`LeakTesting`の設定を有効にするだけで、リークが検出されるとテストが失敗するようになる。

```dart
testWidgets('my widget disposes properly', (tester) async {
  await tester.pumpWidget(const MyWidget());
  // ... テスト処理
}, leakTrackingTestConfig: LeakTrackingTestConfig.debug());
```

これはコードレビューやdevツールで目視するより確実だ。「リークはないはずだ」ではなく「テストが通ればリークはない」という信頼の仕組みを作れる。CI/CDパイプラインにleak_trackerを組み込むことで、リグレッションも防げる。

## 「メモリを意識する」設計の考え方

メモリリークへの対処は「バグを直す」という事後対応だけでなく、「リークしにくい設計を選ぶ」という前提から始まる。

重要な視点をいくつか挙げると:

**オブジェクトの生存期間をUIの生存期間に合わせる**: `StatefulWidget`の`dispose()`を正しく実装するか、状態管理ライブラリに委ねる。Riverpodの`Provider`は参照カウントによる自動破棄をサポートしており、「誰も使っていないProviderは自動でdisposeされる」という設計が可能だ。

**長寿命オブジェクトへの参照を渡さない**: サービスロケーターやシングルトンにウィジェットや`BuildContext`を渡さない。代わりにコールバックをイベントバスやStreamで疎結合にする。

**大きなデータをキャッシュするときはサイズ上限を設ける**: 画像キャッシュや計算結果のキャッシュが際限なく増えることもリークの一形態だ。`LinkedHashMap`を使ってLRUキャッシュを実装するか、既存のパッケージを活用する。

## 計測 → 仮説 → 修正 → 再計測のサイクル

メモリ最適化で陥りがちな罠は「計測せずに直す」ことだ。直感でコードを書き換えても、メモリグラフが改善しているか確認しなければ、本当に直ったかわからない。

正しいアプローチは:

1. DevToolsで怪しい操作を繰り返し、メモリの床が上昇するか確認する
2. スナップショット差分で増えているオブジェクトクラスを特定する
3. そのクラスが生成・破棄されるコードパスを追う
4. 修正を加えて再度1から確認する

このサイクルを回すことが、メモリリークの確実な解決につながる。

## おわりに

メモリリークはアプリの信頼性を静かに蝕む。クラッシュほど目立たないからこそ、設計段階から「dispose()する責務はどこにあるか」を意識し、leak_trackerのような仕組みで自動検出できる体制を整えることが重要だ。DevToolsのメモリプロファイラはリーク検出の強力な武器だが、最終的には「リークしにくいアーキテクチャを選ぶ」という設計判断が最大の防衛線になる。
