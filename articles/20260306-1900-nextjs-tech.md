## はじめに：Next.jsのデプロイ先という選択

Next.jsをどこにデプロイするかは、アプリケーションのパフォーマンス・コスト・機能に大きく影響する設計判断だ。Vercel はもちろんファーストパーティの選択肢だが、Cloudflare Workers（および Pages）もNext.jsの実行環境として現実的な選択肢になってきた。

Cloudflare Workers の特徴は「エッジでコードが動く」こと。ユーザーの物理的な位置に近いデータセンターでコードが実行されるため、コールドスタートが極めて小さく、レイテンシを最小化できる。しかし「エッジで動く」ということは、通常のNode.js環境とは異なる制約があることも意味する。

この記事では、Next.jsをCloudflare Workers上で動かすときの設計上の判断ポイントを、アーキテクチャの視点から整理する。

---

## Cloudflare WorkersとNext.jsの統合：@opennextjs/cloudflare

Next.jsをCloudflare Workers上で動かすためのアダプターとして、`@opennextjs/cloudflare`（OpenNext）が存在する。これはVercelのデプロイ最適化とは独立した、コミュニティドリブンのランタイムアダプターだ。

OpenNextがやっていることを一言で言えば、「Next.jsのビルド出力をCloudflare Workersが理解できる形に変換する」ことだ。App Router のサーバーコンポーネント、サーバーアクション、ルートハンドラーなどが、Workersの環境で動くように変換される。

```bash
# セットアップのイメージ
npm create cloudflare@latest my-next-app -- --framework=next
npx @opennextjs/cloudflare build
wrangler deploy
```

設定ファイル（`wrangler.toml`）でKV、D1、R2などCloudflareのサービスをバインドできるため、データベースやストレージもCloudflareのエコシステムで完結させやすい。

## エッジランタイムの制約を理解する

Cloudflare Workersで動くコードは、V8エンジン上で動作するが、**Node.jsのAPIには制限がある**。これが最も重要な制約だ。

### 使えないNode.js API

- `fs`（ファイルシステム）
- `child_process`
- `net`、`tls`（低レベルのネットワーク）
- バイナリネイティブモジュール（`.node`ファイル）

### 代替手段

| Node.js API | エッジでの代替 |
|------------|--------------|
| `fs` | Cloudflare R2（ファイルストレージ） |
| `crypto` | Web Crypto API（標準API） |
| `Buffer` | `Uint8Array` + TextEncoder |
| PostgreSQL driver | Cloudflare D1、または Hyperdrive |

既存のNext.jsアプリをCloudflare Workersに移行する場合、最大の障壁はこの「Node.jsへの依存」だ。サードパーティライブラリが内部でNode.js固有のAPIを使っていると、そのまま動かない。

## 設計上の判断：どの処理をエッジに置くか

Cloudflare Workersに移行する際の設計判断で最も重要なのは「**何をエッジに置いて、何をオリジン（別のサーバーやAPI）に置くか**」だ。

### エッジに置くと効果的なもの

**認証・認可チェック**  
JWTの検証やセッションの確認は、エッジで完結できる処理の典型例だ。オリジンへのリクエストを発生させる前に、エッジで401を返せる。

**A/Bテストとパーソナライゼーションのルーティング**  
cookieやヘッダーを見て、どのバリアントをレンダリングするかをエッジで決める。これにより、ユーザーが適切なコンテンツを受け取るまでのレイテンシを削減できる。

**静的・準静的なコンテンツのレンダリング**  
SSGやISRに相当するコンテンツは、Cloudflare CDNのキャッシュと組み合わせることで非常に効率的に配信できる。

### オリジンに残すべきもの

**長時間実行が必要な処理**  
Cloudflare WorkersにはCPU時間の制限がある（無料プランは10ms、有料プランでも制限はある）。動画処理・画像変換・複雑な計算はワーカーから別のサービスに委譲する。

**Node.js固有の依存が強いライブラリ**  
Prismaなどの一部ORMはWorkersで動かないか、動かすために特別な設定が必要だ。Cloudflare D1とHyperdriveを使ってデータアクセスを設計し直すか、オリジンにAPIサーバーを残す構成を取る。

## Cloudflare D1とHyperdriveの活用

データベースアクセスはエッジデプロイにおいて最大の課題の一つだ。エッジのサーバーはユーザーの近くにあるが、データベースは一ヶ所にあることが多い。エッジからデータベースへの接続に高レイテンシが生じると、エッジの恩恵が失われる。

**Cloudflare D1**  
SQLiteベースのサーバーレスデータベース。Workersと同じエッジに近い場所でデータを保持できる。小〜中規模のデータ、読み込みが多いワークロードに向いている。

**Cloudflare Hyperdrive**  
既存のPostgreSQLなどへの接続をCloudflareのネットワーク経由でプールする仕組み。Workersからデータベースへの接続コストを削減できる。既存のデータベースを変えずにWorkersを採用したいケースに有効だ。

---

## Cloudflare KVとキャッシュ設計

Cloudflare KV（Key-Value Store）は、全エッジにレプリケーションされる分散KVストレージだ。最終的一貫性（Eventual Consistency）モデルを採用しており、書き込みは数秒〜数分で全エッジに伝播する。

Next.jsのISR（Incremental Static Regeneration）に相当するキャッシュ戦略を設計するとき、KVを使ってキャッシュを保存し、有効期限後に再生成する仕組みを作ることができる。ただし、最終的一貫性のため「キャッシュを即時に全削除する」操作は苦手だという特性を理解した上で設計する必要がある。

---

## Vercel vs Cloudflare：どちらを選ぶか

この問いに唯一の正解はないが、判断の軸を整理するとこうなる。

**Cloudflare Workersが向いているケース**
- グローバルに分散したユーザーへの低レイテンシ配信が最優先
- Cloudflareのエコシステム（D1、R2、KV、Durable Objects）で完結させたい
- コストを抑えたい（無料枠が広い）
- Node.js依存が少ない、または取り除ける

**Vercelが向いているケース**
- Next.jsの最新機能（PPR、Server Actions など）をすぐに使いたい
- Prismaなど、Node.js固有のライブラリへの依存が強い
- チームがNext.jsを始めたばかりで、インフラの複雑さを最小化したい
- Enterprise機能（チーム管理、アナリティクス、セキュリティ）が必要

---

## まとめ

Next.jsをCloudflare Workersで動かすことは、技術的に十分に可能になってきた。しかし「エッジで動く」という環境の制約を理解せずに移行すると、Node.js依存の問題に多くの時間を費やすことになる。

設計の鍵は「何をエッジに置き、何をオリジンに残すか」の判断だ。認証・ルーティング・静的コンテンツ配信はエッジと相性が良く、重い計算・Node.js依存の強い処理はオリジンに残す。

Cloudflareのエコシステム（D1、KV、R2、Hyperdrive）を組み合わせることで、データアクセスを含むアーキテクチャ全体をエッジに最適化した設計が可能になる。コストとパフォーマンスのトレードオフを理解した上で、プロジェクトの要件に合わせた選択をしよう。
