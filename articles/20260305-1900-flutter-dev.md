## はじめに

Flutterアプリにローカルデータ永続化を実装するとき、多くの開発者はまず`shared_preferences`や`hive`を選ぶ。シンプルなキーバリューストアとして手軽だからだ。

しかし、データが複雑になり、リレーションが必要になり、クエリが増えてくると、「やっぱりSQLiteが必要だった」となる瞬間がある。そこで登場するのが**Drift**（旧Moor）だ。

DriftはDart・Flutter向けのSQLiteラッパーだが、単なるラッパー以上の価値を持つ。**Dartのコード生成を使い、型安全なSQLiteアクセスを実現する**——これがDriftの本質だ。

---

## なぜ生のsqliteやsqfliteではなくDriftなのか

`sqflite`はFlutterでSQLiteを使う最もポピュラーな選択肢だ。しかし、sqfliteには重要な問題がある。

SQLクエリを**文字列として書く**ことだ。

```dart
// sqfliteでの典型的なクエリ
final List<Map<String, dynamic>> maps = await db.query(
  'todos',
  where: 'id = ?',
  whereArgs: [id],
);
```

これは「動く」コードだが、型安全ではない。カラム名を誤タイプしても、コンパイル時にエラーにならない。返ってくるのは`Map<String, dynamic>`で、実際にどんなフィールドがあるかは実行時まで分からない。

Driftはこの問題を解決する。テーブル定義をDartのクラスで書くと、コード生成によって**型安全なクエリAPIが自動生成される**。カラム名はDartのプロパティとして参照でき、返り値も具体的な型を持つ。

---

## Driftのテーブル定義とコード生成の仕組み

Driftでは、テーブル定義をDartクラスとして書く。

```dart
import 'package:drift/drift.dart';

// テーブル定義
class Todos extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get title => text().withLength(min: 1, max: 100)();
  BoolColumn get completed => boolean().withDefault(const Constant(false))();
  DateTimeColumn get createdAt => dateTime().nullable()();
}
```

この定義から、`drift_dev`パッケージがコード生成を行うと、`Todo`というデータクラスと、型安全なクエリメソッドが生成される。

データベースクラスの定義：

```dart
@DriftDatabase(tables: [Todos])
class AppDatabase extends _$AppDatabase {
  AppDatabase(QueryExecutor e) : super(e);

  @override
  int get schemaVersion => 1;

  // 型安全なクエリはここに書く
  Future<List<Todo>> getAllTodos() => select(todos).get();

  Stream<List<Todo>> watchIncompleteTodos() => (select(todos)
    ..where((t) => t.completed.equals(false)))
    .watch();
}
```

`where`節で指定するのは文字列ではなく、`t.completed`というDartのプロパティだ。これがDriftの型安全の核心だ。

## Driftの設計思想：宣言的にデータ操作を表現する

Driftの魅力は型安全だけではない。**データ操作を宣言的に記述できる**点も大きな価値だ。

例えば、ウォッチャー（`Stream`）でリアルタイムにDBの変更を監視できる：

```dart
// データが変更されるたびに自動でUIが更新される
final stream = db.watchIncompleteTodos();
StreamBuilder<List<Todo>>(
  stream: stream,
  builder: (context, snapshot) {
    // ...
  },
);
```

これはRiverpodやBlocと組み合わせると非常に強力だ。DBの変更をUIが直接観察できるため、状態管理の複雑さが大きく減る。

## マイグレーション設計：スキーマの変更を安全に行う

ローカルDBを持つアプリの最大の悩みは**スキーマのマイグレーション**だ。アプリのバージョンアップでテーブルを変更したとき、既存ユーザーのデータを安全に移行する必要がある。

Driftはマイグレーションのサポートが充実している。

```dart
@DriftDatabase(tables: [Todos])
class AppDatabase extends _$AppDatabase {
  // ...

  @override
  int get schemaVersion => 2; // バージョンを上げる

  @override
  MigrationStrategy get migration => MigrationStrategy(
    onUpgrade: (migrator, from, to) async {
      if (from < 2) {
        // バージョン1→2: priorityカラムを追加
        await migrator.addColumn(todos, todos.priority);
      }
    },
  );
}
```

Driftには`drift_dev`の`--generate-migration`機能もあり、スキーマの変更を検出して雛形を生成してくれる。手書きのSQLマイグレーションより安全で、変更の追跡が容易だ。

## テスト可能な設計にする

Driftの大きな強みの一つが**テストのしやすさ**だ。

データベースのテストでは通常、実際のファイルI/Oを避けるためにインメモリDBを使いたい。Driftはこれをサポートしている：

```dart
// テスト用にインメモリDBを使う
AppDatabase createTestDatabase() {
  return AppDatabase(NativeDatabase.memory());
}
```

テストコードでは本番と全く同じAPIを使いつつ、実際のファイルには書き込まない。DIを組み合わせれば、テストと本番でDBの実装を差し替えられる。

## sqflite・Hive・Isarとの比較

どのライブラリを選ぶべきか、状況別に整理する：

**キーバリュー的な単純なデータ保存**
→ `shared_preferences`または`Hive`。Driftはオーバーエンジニアリングになる。

**複雑なリレーションなしの構造化データ**
→ `Isar`も有力な選択肢。Dartオブジェクトをそのまま永続化でき、セットアップが簡単。

**リレーショナルなデータ、複雑なクエリ、既存のSQLへの親しみ**
→ `Drift`。コード生成のセットアップコストはあるが、長期的な保守性が高い。

**sqfliteとDrift**を比較すると、sqfliteはシンプルさが魅力だが、型安全性とマイグレーション管理はDriftが圧倒的に優れる。アプリが成長することを想定しているなら、最初からDriftを選ぶ方が賢明だ。

---

## Driftを使う上での注意点

**コード生成のビルドが必要**

Driftはコード生成に依存するため、テーブル定義を変更するたびに`dart run build_runner build`を実行する必要がある。これを忘れると古い定義と実装の乖離が発生する。`watch`モードを活用して開発中は自動生成しておくと楽だ。

**学習コスト**

SQLに慣れている人ならDriftのAPIは直感的だが、SQLに不慣れな人には概念の学習コストがある。また、公式ドキュメントは英語のみで、日本語情報はまだ少ない。

**バンドルサイズ**

コード生成のため、小さなアプリには重すぎることもある。データの複雑さに応じて選択すべきだ。

---

## まとめ：ローカルDBに「型安全」をもたらす価値

DriftがFlutterのSQLiteアクセスにもたらす本質的な価値は、型安全性と宣言的なAPI設計だ。

生のSQLを文字列で書くことのリスク——タイプミス、スキーマ変更時の取り残し、実行時まで見えないエラー——これらをコンパイル時に検出できる仕組みがDriftにはある。

ローカルデータベースは多くのFlutterアプリの心臓部だ。その部分を「動くかもしれない文字列」で管理するより、Dartの型システムの恩恵を最大限に受けた設計にする方が、長期的な保守性は大きく向上する。データの複雑さが増してきたとき、Driftへの投資は確実に報われる。
