## セキュリティは「最後に確認するもの」ではない

Webアプリケーション開発においてセキュリティテストが後回しになりやすい理由は、心理的な難しさにある。機能テストは「動くか動かないか」がすぐにわかる。しかしセキュリティの問題は「動いているが脆弱だ」という状態が長い間見えないまま存在し続ける。

かつての開発フローでは、リリース直前にペネトレーションテスト（侵入テスト）を外部委託し、指摘された問題を修正するというプロセスが一般的だった。このアプローチの問題は、問題の発見が遅すぎることだ。アーキテクチャレベルの脆弱性がリリース直前に発覚しても、修正のコストと影響範囲が大きすぎて手をつけられないという状況が生まれる。

DevSecOpsという考え方は、セキュリティを開発プロセスの最初から組み込む（Shift Left）発想だ。セキュリティテストをCI/CDパイプラインに統合し、開発の各ステップでフィードバックを得ることで、問題を小さいうちに潰せる。本記事では、Webアプリケーション開発のQAプロセスにセキュリティテストを組み込む考え方と手法を整理する。

## セキュリティテストの種類を理解する

セキュリティテストは手法によって特性が大きく異なる。何を目的に、どのフェーズで使うかを整理しておく必要がある。

**SAST（Static Application Security Testing）**

ソースコードを実行せずに静的解析し、脆弱性のパターンを検出する。SQLインジェクション、クロスサイトスクリプティング（XSS）、ハードコードされた認証情報など、コードレベルで特定できる問題に向く。

代表的ツールとしてSemgrep（ルールベースの静的解析、OSS）、Snyk Code（開発者向けSAST）がある。

**DAST（Dynamic Application Security Testing）**

実行中のアプリケーションに対してリクエストを送り、脆弱性を動的に検出する。実際のHTTPリクエスト・レスポンスを扱うため、SASTでは検出できないランタイムの問題（設定ミスによるセキュリティヘッダーの欠落など）も見つけられる。

OWASP ZAP（OSS）、Burp Suite（商用）が代表的なDASTツールだ。

**SCA（Software Composition Analysis）**

使用している依存ライブラリの既知の脆弱性（CVE）を検出する。`npm audit`や`snyk test`がこれにあたる。パッケージの更新だけで防げる脆弱性を見落とさないための基本的な検査だ。

**ペネトレーションテスト（手動）**

熟練したセキュリティエンジニアが実際に攻撃者の視点でシステムを調査する。自動化ツールでは検出できない複合的な脆弱性や、ビジネスロジックの問題を見つけるのに有効だ。自動化ツールの補完として位置づけ、重要リリース前や定期的に実施する。

## OWASP Top 10 を開発チームの共通語にする

セキュリティの知識を開発チーム全体に広げるための出発点として、OWASP Top 10（Open Worldwide Application Security Project）が役立つ。Webアプリケーションで特に重要な脆弱性カテゴリをリスト化したもので、数年ごとに更新される。

2021年版では以下のカテゴリが上位に並ぶ。

1. アクセス制御の不備
2. 暗号化の失敗
3. インジェクション（SQLi、XSSなど）
4. 安全でない設計
5. セキュリティの設定ミス
6. 古いコンポーネントの使用
7. 識別と認証の失敗
8. ソフトウェアとデータの整合性の失敗
9. ログとモニタリングの失敗
10. サーバーサイドリクエストフォージェリ（SSRF）

これらを開発チームの共通語にするだけで、コードレビューの質が変わる。「この実装はインジェクションのリスクがないか」「認証チェックが漏れていないか」という観点がレビューに自然に組み込まれるようになる。

## CI/CDへの組み込み方

セキュリティテストをパイプラインに入れるとき、「何をどのタイミングで実行するか」の設計が重要だ。すべてのチェックをすべてのコミットで実行すると、フィードバックループが遅くなりすぎる。

**コミット・プルリクエスト時（高速・軽量）**
- SCA（npm audit、Snyk）でライブラリの既知脆弱性を確認
- SASTの簡易スキャン（Semgrep）でコードの問題パターンを検出
- シークレットのスキャン（git-secrets、gitleaksなど）でAPIキーやパスワードのコミットを防ぐ

**ステージング環境デプロイ後（より重いチェック）**
- DASTによる動的スキャン
- セキュリティヘッダーの検査
- TLS/SSL設定の確認

**定期実行（週次・月次）**
- 依存ライブラリの脆弱性の定期スキャン
- より深いSASTスキャン

```yaml
# GitHub Actions での簡易セキュリティスキャン例
- name: Run npm audit
  run: npm audit --audit-level=high

- name: Run Semgrep
  uses: semgrep/semgrep-action@v1
  with:
    config: "p/owasp-top-ten"
```

## シークレット管理は特別に扱う

セキュリティ問題の中でも「APIキーやパスワードのソースコードへの混入」は特に防ぎやすく、かつ頻発する問題だ。gitleaksやgit-secretsをpre-commitフックとCIの両方に仕込むことで、シークレットがリポジトリに入る前に検出できる。

一度シークレットがpublicリポジトリにpushされると、たとえ即座に削除しても漏洩したとみなすべきだ。GitHubはシークレットスキャン機能を提供しており、既知のパターンのシークレットが検出されると自動で通知・無効化する仕組みもある。

環境変数と`.env`ファイルの管理も重要だ。`.env.local`を`.gitignore`に必ず入れる、本番のシークレットはCI/CDのSecrets機能やVaultに格納するという基本的な習慣がほとんどの問題を防ぐ。

## セキュリティヘッダーのテスト

WebアプリケーションのHTTPレスポンスヘッダーには、ブラウザに「このページをどう扱うか」を指示するセキュリティ設定がある。これらが正しく設定されているかをテストすることは、比較的コストが低く効果が高い。

確認すべき主なヘッダー：
- `Content-Security-Policy`（XSSの緩和）
- `Strict-Transport-Security`（HTTPSの強制）
- `X-Content-Type-Options: nosniff`（MIMEタイプスニッフィングの防止）
- `X-Frame-Options`（クリックジャッキングの防止）
- `Referrer-Policy`（リファラー情報の制限）

securityheaders.comのようなツールで本番URLを検査するか、CIでHTTPリクエストを投げてレスポンスヘッダーをアサートするテストを書くことで継続的に確認できる。

## 「ゼロリスク」は目標にならない

セキュリティテストをQAに組み込むとき、「すべての脆弱性を取り除く」を目標にするとプロセスが停滞する。自動スキャンツールは多数のFalse Positive（誤検知）を出す。重要性のないアラートをすべて消し込もうとすると、本当に重要な問題への集中力が失われる。

リスクベースのアプローチが現実的だ。検出された問題を重大度（Critical/High/Medium/Low）で分類し、Critical/HighはCIを止める、MediumはIssueを作る、Lowは定期的に見直す、という基準を設けることで、チームが動きやすくなる。

セキュリティテストの目的は「完璧なシステムを作る」ではなく「攻撃の費用対効果を攻撃者にとって割に合わないほど高くする」ことだ。そのためには、すべての脆弱性を塞ぐよりも、高リスクな部分を継続的に監視・改善するプロセスを維持する方が効果的だ。

## まとめ

セキュリティテストをQAプロセスに組み込む本質は、「発見を早くする」ことだ。SAST、DAST、SCAを適切なタイミングでCI/CDに配置し、シークレットのスキャンとセキュリティヘッダーの検査を習慣化する。OWASP Top 10を開発チームの共通語にすることで、コードレビューにセキュリティの視点が自然に入るようになる。完璧なセキュリティを求めるのではなく、リスクベースで優先度をつけて継続的に改善するプロセスを維持することが、実践的なDevSecOpsの基盤だ。
