## GraphQLのテストが「RESTより難しい」理由

REST APIのテストは比較的シンプルだ。エンドポイントごとにHTTPメソッドとレスポンスを確認すれば、大まかな動作保証ができる。

しかしGraphQLになると話が変わる。**エンドポイントは1つ（`/graphql`）だが、クエリの組み合わせは無限に近い**。フロントエンドが「どのフィールドを要求するか」を動的に決めるGraphQLの柔軟性は、テストカバレッジの考え方を根本から変える必要がある。

さらに、GraphQLにはスキーマという「型の契約」が存在する。この契約を軸にしたテスト設計が、品質保証の中心になる。

## GraphQLテストの3層構造

GraphQLアプリケーションのテストは、次の3層で考えると整理しやすい。

### 1. スキーマテスト（Schema Testing）

スキーマそのものが仕様だ。型の変更・フィールドの削除・必須制約の変化は、すべてのクライアントに影響する破壊的変更になりうる。

**スキーマのスナップショットテスト**を導入することで、意図しないスキーマ変更を検出できる。

```typescript
import { printSchema, buildSchema } from 'graphql';
import schema from '../src/schema';

test('スキーマが変更されていないこと', () => {
  const schemaString = printSchema(schema);
  expect(schemaString).toMatchSnapshot();
});
```

スナップショットが変わったとき、それが意図した変更かどうかをレビューで確認する。「スキーマはコード」という意識を持ち、変更は意図的に、記録として残す。

### 2. リゾルバテスト（Resolver Testing）

リゾルバはGraphQLの「ビジネスロジックが宿る場所」だ。各フィールドのデータ取得・変換・バリデーションをリゾルバ単位でテストする。

リゾルバテストのポイントは、**GraphQL層とビジネスロジック層を分離すること**。リゾルバをスリムに保ち、ロジックはサービス層やドメイン層に切り出すと、テストしやすくなる。

```typescript
// ユーザーリゾルバのテスト
describe('User Resolver', () => {
  it('ユーザーIDからユーザー情報を取得できる', async () => {
    const mockUserService = {
      findById: jest.fn().mockResolvedValue({
        id: '1',
        name: 'テストユーザー',
        email: 'test@example.com'
      })
    };

    const result = await userResolver.Query.user(
      null,
      { id: '1' },
      { userService: mockUserService }
    );

    expect(result.name).toBe('テストユーザー');
    expect(mockUserService.findById).toHaveBeenCalledWith('1');
  });
});
```

### 3. 統合テスト（Integration Testing）

実際のクエリを投げて、エンドツーエンドで動作を確認するテスト。Apolloなら `ApolloServer` のテスト用メソッドを使って、本物に近い環境でテストできる。

```typescript
import { ApolloServer } from '@apollo/server';
import { startStandaloneServer } from '@apollo/server/standalone';

const { url } = await startStandaloneServer(server, {
  listen: { port: 0 } // ランダムポートでテスト用起動
});

const response = await fetch(url, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    query: `
      query GetUser($id: ID!) {
        user(id: $id) {
          id
          name
          posts { title }
        }
      }
    `,
    variables: { id: '1' }
  })
});
```

## スキーマ駆動のコントラクトテスト

GraphQLの強みは「スキーマが契約書」であることだ。この契約を中心に、フロントエンドとバックエンドのテストを連携させる考え方が**スキーマ駆動コントラクトテスト**だ。

### Breaking Change検出

スキーマ変更が破壊的かどうかを自動判定するツールとして、`@graphql-inspector/core` が有用だ。

```bash
graphql-inspector diff old-schema.graphql new-schema.graphql
```

フィールドの削除・型の変更・引数の必須化などが「破壊的変更」として検出される。CI/CDパイプラインに組み込むことで、誰かがスキーマを壊してもマージ前に気づける。

### クライアント側のクエリバリデーション

バックエンドのスキーマに対して、フロントエンドが使っているクエリが有効かどうかを静的チェックする仕組みも作れる。`graphql-codegen` や `graphql-eslint` を活用することで、「存在しないフィールドを要求している」「型が合っていない」といったミスを開発時に検出できる。

## GraphQL特有のテスト課題

### N+1問題のテスト

GraphQLのリレーションをnaiveに実装すると、N+1クエリ問題が発生しやすい。DataLoaderでバッチ処理しているが、テストでそれが正しく機能しているか確認することが重要だ。

テスト時に実際に発行されたSQLクエリ数を計測して、期待するバッチ数と一致しているか確認するアプローチが使える。ORMのクエリログをカウントする仕組みを用意しておくと、N+1のリグレッションを検出できる。

### ページネーションのテスト

Relay-style cursorページネーション（`edges`/`nodes`/`pageInfo`）は、実装パターンが複雑になりがちだ。

- `hasNextPage` / `hasPreviousPage` のフラグが正しいか
- `cursor` が正確に次のページを指しているか
- 境界値（最初のページ・最後のページ・1件のみ）の動作

これらをテストケースとして網羅しておかないと、本番でページング不全が起きる。

### エラーハンドリングのテスト

GraphQLは成功時も失敗時も**HTTPステータスは200で返す**ことが多い。エラーはレスポンスボディの `errors` フィールドに入る。

```json
{
  "data": null,
  "errors": [
    {
      "message": "Unauthorized",
      "locations": [{ "line": 2, "column": 3 }],
      "path": ["user"]
    }
  ]
}
```

テストでHTTPステータスコードだけを確認していると、GraphQLのエラーを見落とす。`errors` フィールドの有無と内容を必ず確認するように習慣化する。

## モックとテスト環境の設計

フロントエンドのGraphQLテストでは、サーバーへの実際のリクエストをモックすることが多い。`msw`（Mock Service Worker）はGraphQL用のハンドラも持っており、テスト環境での活用が進んでいる。

```typescript
import { graphql, HttpResponse } from 'msw';

const handlers = [
  graphql.query('GetUser', () => {
    return HttpResponse.json({
      data: {
        user: {
          id: '1',
          name: 'テストユーザー',
          email: 'test@example.com'
        }
      }
    });
  })
];
```

mswの利点は、**テスト環境とブラウザ開発環境で同じモックハンドラを使いまわせること**だ。本物のネットワーク処理をインターセプトするため、クライアント側のコードをテスト用に書き換える必要がない。

## 品質ゲートとしてのスキーマ管理

チームで開発する場合、GraphQLスキーマの変更管理を「品質ゲート」として機能させることが重要だ。

**CIでやるべきこと：**

1. **スキーマのスナップショット差分確認**：変更が意図的かレビューで確認
2. **Breaking Change検出**：破壊的変更はマージブロック
3. **全クエリのバリデーション**：フロントエンドのクエリがスキーマに準拠しているか
4. **リゾルバの単体テスト**：ビジネスロジックのカバレッジを維持
5. **統合テスト**：主要なクエリ・ミューテーションの動作確認

スキーマを「生きた仕様書」として扱い、その変更をテストで守る——これがGraphQLにおけるQAの本質だ。

---

## まとめ

GraphQLのテストは「エンドポイントが1つだから簡単」ではなく、「クエリの自由度が高いから複雑」だ。スキーマという強力な型システムをテストの軸として活用し、スキーマテスト・リゾルバテスト・統合テストの3層を整備することで、変更に強い品質保証の仕組みが作れる。

大切なのは、GraphQLの「スキーマは契約だ」という思想をテスト設計にも反映させることだ。その契約を守る仕組みをCIに組み込むことで、フロントエンドとバックエンドの連携が安定したものになる。
