## はじめに：テストと仕様書が別物である問題

多くのプロジェクトで、テストコードと仕様書は別々に存在している。仕様書はドキュメントとして書かれ、テストコードはエンジニアが別途実装する。そして時間が経つにつれ、両者の間にズレが生まれ始める。

「仕様書には書いてあるが、テストが通っている」「テストは全部グリーンだが、動きが仕様と違う」——こうした状況は珍しくない。

BDD（Behavior-Driven Development）は、この根本的な問題に対する一つの答えだ。テストを「動く仕様書」として捉え、技術者と非技術者が共通の言語でシステムの振る舞いを定義しようという思想を持つ。本記事ではBDDの設計思想と、それを実現するGherkin・Cucumberというツールの本質を探る。

## BDDとは何を解こうとしているか

BDDはTDD（テスト駆動開発）から派生して生まれたが、重点が異なる。

TDDが「コードを正しく書けているか」という技術的な検証に主眼を置くのに対し、BDDは「正しいものを作っているか」というビジネス価値の検証に焦点を当てる。この区別は小さいように見えて、実は大きな設計上の差異につながる。

BDDの創始者Dan Northは、TDDを実践する中で「何をテストすべきか」「テストをどう命名するか」という問いに多くの開発者が悩むことに気づいた。そして、その解決策として「振る舞い（behavior）を中心に考える」というアプローチを提唱した。

システムの振る舞いは、利用者の立場から「ある状況で、こういう操作をしたとき、こうなる」という形で自然に表現できる。この表現をそのままテストとして実行可能にしようというのがBDDの核心だ。

## Gherkinという共通言語

BDDを実現するために設計された言語がGherkinだ。Gherkinは特定の構文規則を持つ自然言語に近い記述形式で、英語・日本語を含む多くの言語でシナリオを書ける。

基本的な構造は `Given-When-Then` という三部構成だ。

```gherkin
Feature: ユーザーログイン

  Scenario: 正しい認証情報でのログイン
    Given ユーザーが登録済みのメールアドレスを持っている
    When ユーザーが正しいメールアドレスとパスワードを入力して送信する
    Then ユーザーはダッシュボード画面にリダイレクトされる
    And ウェルカムメッセージが表示される

  Scenario: 誤ったパスワードでのログイン試行
    Given ユーザーが登録済みのメールアドレスを持っている
    When ユーザーが正しいメールアドレスと誤ったパスワードを入力して送信する
    Then エラーメッセージが表示される
    And ユーザーはログインページに留まる
```

`Given` は前提条件、`When` はアクション、`Then` は期待される結果を表す。`And` と `But` は前のステップと同じ種類の継続として使える。

この記述は、プロダクトマネージャーも読める。エンジニアも読める。QAエンジニアも読める。この共通言語性こそがGherkinの価値だ。

## Cucumberとステップ定義

Gherkinで書かれたシナリオを実際に実行可能なテストにするのがCucumberだ（JavaScriptエコシステムではCucumber.js、Ruby向けの元祖Cucumber、Java向けのCucumber-JVMなど複数の実装がある）。

Cucumberはシナリオの各ステップを、実際のコードと紐付ける仕組みを提供する。この紐付けを「ステップ定義」と呼ぶ。

例えば先ほどのシナリオを実行するために、次のようなステップ定義を書く。

```javascript
const { Given, When, Then } = require('@cucumber/cucumber');

Given('ユーザーが登録済みのメールアドレスを持っている', async function () {
  await createUser({ email: 'test@example.com', password: 'secure123' });
});

When('ユーザーが正しいメールアドレスとパスワードを入力して送信する', async function () {
  await this.page.fill('[data-testid="email"]', 'test@example.com');
  await this.page.fill('[data-testid="password"]', 'secure123');
  await this.page.click('[data-testid="login-button"]');
});

Then('ユーザーはダッシュボード画面にリダイレクトされる', async function () {
  await this.page.waitForURL('/dashboard');
});
```

Gherkinのステップとステップ定義は正規表現やプレーンテキストマッチで紐付けられる。このため、同じステップ定義を複数のシナリオで再利用できる。

## シナリオをどう設計するか

BDDの実践で多くのチームが躓くのが、シナリオの粒度と表現の問題だ。

よくある失敗パターンは、UIの操作詳細を仕様に書き込みすぎることだ。

```gherkin
# ❌ UIに依存しすぎた書き方
When ユーザーが右上の青いボタンをクリックする

# ✅ ビジネス的な意図を表現する書き方
When ユーザーがアカウント設定を開く
```

前者はUIが変わるたびにシナリオを書き直す必要が生まれる。後者はビジネス上の意図を表現しており、UIの実装詳細から独立している。

設計の指針は「なぜ」ではなく「何を」だ。ユーザーが何をしようとしているかを記述し、どうやって達成するかはステップ定義に任せる。

また、シナリオの数も注意が必要だ。すべての組み合わせをシナリオとして書き出すと、膨大な数になってしまう。境界値や異常系は `Scenario Outline` でパラメータ化して表現し、本質的な振る舞いの数を絞るのが実用的なアプローチだ。

## BDDは「プロセス」である

BDDを単なるツールとして捉えると、本質を見失う。Cucumberを導入することがBDDではない。

BDDの本質は、「仕様を定義するセッション」を持つというプロセスにある。Dan Northや同僚のChris Mattsが提唱した「スリーアミーゴス（Three Amigos）」というプラクティスがある。これは、開発者・テスター・ビジネス代表者の3者が一緒に座り、シナリオを書くというものだ。

このセッションで起きることが重要だ。「この場合はどうなりますか？」「それはバグですか？仕様ですか？」「このシナリオは現実的ですか？」——これらの問いを早い段階で共有することで、実装後に発見されていたはずの問題を事前に潰せる。

Gherkinのシナリオは、その議論の成果物だ。コードを書く前に、振る舞いについて合意が取れているという証拠として機能する。

## コストとベネフィットの現実

BDDはすべてのプロジェクトに適しているわけではない。正直に言えば、維持コストは高い。

Gherkinシナリオはコードと同様に変更のたびにメンテナンスが必要だ。UIの変更、仕様変更のたびにシナリオとステップ定義を更新しなければならない。ビジネス側の参加がなければ、BDDの最大の価値である「共通言語」が失われ、ただ書き方が冗長なE2Eテストになってしまう。

BDDが最も価値を発揮するのは、ビジネスルールが複雑で、技術者と非技術者が密に協業する必要があるドメインだ。金融サービス、医療、EC——「この条件のとき、正確にこう動くべき」という仕様が詳細に定義されている領域では、BDDの表現力が強みになる。

逆に、プロトタイプ段階や頻繁に仕様が変わるプロダクトでは、Gherkinの維持コストが重荷になることも多い。

## おわりに

BDDが問いかけるのは、「テストを誰のために書くか」という根本的な問いだ。

コードの正しさを確認するためだけなら、ユニットテストや統合テストで十分かもしれない。しかし、ビジネスの意図とシステムの振る舞いが一致しているかを継続的に確認したいなら、BDDのアプローチは強力な手段になる。

Gherkinの構文を覚えることより、「誰がシナリオを読み、誰が書くか」というプロセスの設計の方が、BDD導入の成否を分ける。ツールは思想を実現するための手段に過ぎない。
