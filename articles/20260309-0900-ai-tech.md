## 同じプレフィックスを何度も送るコスト

LLMアプリケーションを本番運用していると、ある共通の悩みに直面する。毎回のリクエストに「システムプロンプト」「ドキュメント本文」「参照データ」などの長い文脈を繰り返し送信しているという現実だ。

チャットbotなら会話の冒頭に置くシステムプロンプト、RAGなら検索で取得したドキュメント群、コードレビューツールなら分析対象のファイル全体——これらは同じリクエストのたびにトークンとして消費される。トークン単価が下がっても、リクエスト数が増えれば積み上がるコストは無視できない。

**Prompt Caching**は、この「同じものを何度も送る」という非効率を根本から解決する仕組みだ。

---

## Prompt Cachingの仕組み

仕組みをシンプルに言うと、**プレフィックス（文脈の先頭部分）をサーバー側にキャッシュしておき、次のリクエストでは「ここまでは前回と同じ」と伝えるだけで済む**、というものだ。

AnthropicのClaude APIでは `cache_control` フラグを使って、特定のコンテンツブロックをキャッシュ対象として指定する。

```python
messages = [
    {
        "role": "user",
        "content": [
            {
                "type": "text",
                "text": long_document_text,
                "cache_control": {"type": "ephemeral"}  # ここをキャッシュ
            },
            {
                "type": "text",
                "text": "このドキュメントの要点を教えてください"
            }
        ]
    }
]
```

初回リクエストでは通常通りトークンが消費されるが、2回目以降は**キャッシュヒット**として処理され、入力トークンコストが大幅に削減される（Anthropicの場合、キャッシュヒット時は通常の約10%）。

OpenAIも類似の「Prompt Caching」機能を提供しており、プレフィックスの一致を自動検出してキャッシュを適用する仕組みになっている。

---

## キャッシュが有効に機能する条件

Prompt Cachingは万能ではない。効果を最大化するためには、**キャッシュされる部分が「変化しない」こと**が前提になる。

### プレフィックス安定性の原則

LLMに渡すプロンプトの構造を考えると、次のように分類できる。

1. **静的部分**：システムプロンプト、役割定義、制約条件
2. **半静的部分**：参照ドキュメント、コードベース、データシート
3. **動的部分**：ユーザーの質問、会話履歴、セッション固有のデータ

キャッシュが効くのは1と2の部分だ。重要なのは、**静的・半静的な部分を必ずプロンプトの先頭に配置すること**。動的な部分が先に来てしまうと、プレフィックスが毎回変わってしまいキャッシュがヒットしない。

### キャッシュの有効期限

AnthropicのPrompt Cachingでは、キャッシュの有効期限は約5分（2024年時点）。同じセッション内での連続したリクエストには効果的だが、時間をまたぐ場合はキャッシュが失効する。

ユーザーが長時間セッションを続けたり、非同期的にリクエストが飛んだりする場合は、キャッシュヒット率が期待より下がることを考慮に入れる必要がある。

---

## 設計思想：「何を変えないか」を先に決める

Prompt Cachingを活用するアプリ設計では、考え方を逆転させる必要がある。「何を毎回変えるか」ではなく、「**何を変えないか**」を先に明確にするのだ。

### パターン1：大容量ドキュメントQA

法律文書、技術仕様書、マニュアルなど、大量のドキュメントに対してユーザーが質問するケースを考える。

```
[キャッシュ対象]
- システムプロンプト（QAアシスタントとしての役割定義）
- ドキュメント全文（数万トークン規模）

[毎回変わる部分]
- ユーザーの質問（数十〜数百トークン）
```

ドキュメントの読み込みコストを1度だけ払い、以後の質問は安価にこなせる。RAGで「検索して渡す」アプローチと比べると、検索精度の限界を超えてドキュメント全体をコンテキストに置ける点が強みになる。

### パターン2：コードレビュー・分析ツール

コードベース全体をコンテキストに渡して質問するケースでも有効だ。

```
[キャッシュ対象]
- レビュー基準・コーディング規約（システムプロンプト）
- 対象ファイルの全コード

[毎回変わる部分]
- レビュー観点の指示（「セキュリティ視点で見て」「パフォーマンス改善点は？」）
```

同じファイルを複数の観点でレビューする場合、ファイル読み込みコストは最初の1回だけ。

### パターン3：マルチターンの会話

長い会話を続けるチャットbotでは、会話履歴が積み上がるにつれてコストが増大する。Prompt Cachingを使えば、**既に送信済みの会話部分をキャッシュ対象とし、新しいターンの部分だけを追加**するような設計ができる。

---

## コスト削減の現実的な見積もり

Prompt Cachingの効果は、**キャッシュ対象となるトークン数が全体に占める割合**によって大きく変わる。

仮にシステムプロンプト＋ドキュメントで10,000トークン、ユーザーの質問が100トークン、これを1日1,000回リクエストするケースを考える。

キャッシュなし：10,100トークン × 1,000回 = 10,100,000 入力トークン  
キャッシュあり（ヒット率80%）：10,000 × 0.1 × 800 + 10,100 × 200 ≈ **2,820,000 入力トークン相当**

ざっくり7割以上のコスト削減になりうる。ただし、キャッシュヒット率が実際にどの程度になるかは、アプリのユーザー行動パターンによって異なる。

---

## キャッシュを意識したプロンプト設計の原則

Prompt Cachingを最大限活かすプロンプト設計には、いくつかの原則がある。

**安定した部分を先頭に固める**：前述の通り、変化しない部分を必ずプロンプトの先頭に集める。後半で静的なコンテンツを追加しても、前半が変わっていればキャッシュは無効になる。

**粒度を意識してキャッシュポイントを設定する**：Anthropicのように明示的にキャッシュポイントを設定できるAPIでは、「どこまでをキャッシュするか」の境界を慎重に決める。細かすぎると管理コストが上がり、粗すぎると恩恵が少ない。

**バージョン管理の考え方を持つ**：システムプロンプトを変更するたびにキャッシュが無効になる。頻繁に変えるものと、長期間固定するものを意識的に分離する設計が必要だ。

---

## キャッシュとRAGの使い分け

Prompt Cachingは「ドキュメントをまるごと渡す」アプローチを現実的にするが、だからといってRAGが不要になるわけではない。

RAGの強みは「関連部分だけを選別する」ことで、コンテキスト長を抑え、ノイズを減らすことにある。一方、Prompt Cachingの強みは「全体を渡してもコストを抑えられる」こと。どちらを選ぶかは、**ドキュメントの規模**と**検索精度の要求**によって変わる。

数千トークン規模で全体をコンテキストに入れられるならキャッシュ活用、数十万トークン規模でコンテキスト長の上限に当たるならRAGが依然として有効な選択肢になる。

---

## まとめ

Prompt Cachingは、LLMアプリのコスト最適化における新しい設計軸だ。「毎回全部送る」という素朴な実装から、「変わらない部分は一度だけ払う」という発想への転換が、コストと応答速度の両面で恩恵をもたらす。

重要なのは技術的な仕組みよりも、**アプリケーションの「静的な部分」と「動的な部分」を意識的に切り分ける設計習慣**を持つことだ。Prompt Cachingはその切り分けを経済的なインセンティブとして可視化してくれる、非常に実践的な機能といえる。
