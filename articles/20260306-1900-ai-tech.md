## はじめに：MLOpsという言葉の誤解

「MLOps」という言葉を聞くと、多くのエンジニアはまず「実験のトラッキング」や「モデルのバージョン管理」を思い浮かべる。MLflow や Weights & Biases を使って実験を記録し、どのパラメータが最良だったかを追いかける——そういったツールの話だと受け取られることが多い。

しかし、MLOpsの本質はそこにはない。MLOpsの中心にあるのは「**信頼の仕組みを作ること**」だ。

本番環境で稼働するモデルが、今日も昨日と同じように動いているか。データが変化したとき、モデルの精度はどう変わるか。そのような問いに対して、自動的・継続的に答え続けられる仕組みを構築することこそが、MLOpsの核心である。

---

## なぜ「実験管理」だけでは不十分か

機械学習プロジェクトの多くは、実験フェーズでは成功しても、本番運用に移行した途端に問題が噴出する。その原因として代表的なものを挙げてみよう。

**データドリフト（Data Drift）**  
モデルの訓練時に使ったデータと、本番で受け取るデータの分布がずれていく現象。ユーザーの行動が変わったり、入力元のシステムが変化したりすることで、じわじわと「訓練時のモデル」と「現実」の乖離が進む。

**コンセプトドリフト（Concept Drift）**  
モデルが予測しようとしている「概念」そのものが変わる現象。例えば「優良顧客の定義」が事業戦略の変更によって変化した場合、データの分布が変わらなくても、正解ラベルの意味が変わってしまう。

**訓練・サービングスキュー（Training-Serving Skew）**  
訓練時の前処理ロジックと、本番サービングで使われる前処理が微妙にずれている状態。これは人間のミスから生じることが多く、「実験では良かったのに本番では悪い」という事態を引き起こす。

これらの問題は、実験を丁寧に管理するだけでは防げない。継続的な監視と、問題を検知したときの自動的なフィードバックループが必要になる。

## MLOpsを「パイプライン」として捉える

MLOpsを設計するとき、一番重要な視点は「MLをパイプラインとして見ること」だ。

ソフトウェアエンジニアリングでは、コードの変更が自動的にテストされ、デプロイされる CI/CD パイプラインが当たり前になっている。MLOpsはこれを機械学習に拡張したものだが、機械学習には「コードだけでなく、データとモデルも成果物になる」という特殊性がある。

MLパイプラインが対象とする成果物は3種類ある：

| 成果物 | 変化の契機 | 管理の方法 |
|--------|-----------|-----------|
| コード | 開発者の変更 | Git + CI/CD |
| データ | 外部からの流入 | バージョン管理 + 検証 |
| モデル | 再訓練・ファインチューニング | モデルレジストリ |

この3つを独立して管理するのではなく、**「データが変わればモデルを再訓練し、モデルが変わればサービングを更新する」という連鎖**をパイプラインとして自動化することがMLOpsの狙いだ。

## 設計の要点：4つのステージ

MLOpsパイプラインを設計するとき、以下の4つのステージを意識すると整理しやすい。

### 1. データ検証ステージ

パイプラインの入口に必ず置くべきなのが、データ検証だ。訓練に使うデータが「期待通りの形式・範囲・分布」であるかを自動的にチェックする。

Google の TensorFlow Extended（TFX）や、オープンソースの Great Expectations といったツールを使えば、スキーマの定義・統計量の計算・ドリフトの検出をコードとして記述できる。人間が目でチェックするのではなく、パイプラインが自動的に「このデータは信頼できるか」を判断する。

```python
# Great Expectations のイメージ
context.run_checkpoint(checkpoint_name="training_data_check")
# → データが期待値を満たさない場合はパイプラインを停止
```

### 2. 実験・訓練ステージ

実験管理ツール（MLflow、W&B など）を使って、ハイパーパラメータ・評価指標・使用データセットのバージョンを記録する。ここで重要なのは「再現可能性」だ。6ヶ月後に同じモデルを再現できるよう、実験の全条件をコードと一緒に管理する。

### 3. モデル評価・承認ステージ

新しいモデルを自動的に本番に出すのは危険だ。必ず「現行モデルと比較して、新モデルは本当に優れているか」を評価するステージを挟む。A/Bテストや Shadow Mode（本番トラフィックを使ってテストするが、結果は返さない）で比較評価を行い、人間またはポリシーが承認してから本番へ昇格させる。

### 4. モデル監視ステージ

デプロイ後も継続的に監視する。監視すべき指標は大きく2種類だ：

- **技術的指標**：レイテンシ、エラー率、スループット
- **品質指標**：モデルの予測分布、データドリフトのスコア

品質指標は通常、ラベルがすぐには得られない。ユーザーのフィードバックや遅延ラベリングで事後的に品質を計測する仕組みも設計する必要がある。

## LLMアプリケーションにおけるMLOps

LLMが普及したことで、MLOpsの対象が変わりつつある。自社でモデルを訓練しないケースでも、プロンプトのバージョン管理・評価・ドリフト検知という観点でMLOpsの思想は有効だ。

- **プロンプトのバージョン管理**：プロンプトをコードと同様に Git で管理し、変更の影響を評価する
- **LLM評価パイプライン**：新しいプロンプトや RAG 設定が品質を下げていないかを自動評価
- **出力分布の監視**：モデルのレスポンスが時間とともに変化していないか（モデルプロバイダーのアップデートなどで変わることがある）を監視

「LLMを使えばMLOpsは不要」ではなく、「LLMを使うからこそ、異なる形のMLOpsが必要」という認識が重要だ。

---

## 組織とMLOps：ツールより先に必要なもの

MLOpsを導入しようとするとき、ツール選びよりも先に整理すべき問いがある。

「誰が、何に責任を持つのか？」

モデルの品質に責任を持つのはデータサイエンティストか、MLエンジニアか、プロダクトチームか。本番でモデルが劣化したときに誰がアラートを受け取り、どのアクションを取るのか。

MLOpsは技術的な仕組みである前に、**組織の責任分担を明確にするための枠組み**でもある。ツールを導入する前に、この問いに答えを出しておくことが、MLOpsプロジェクトの成否を大きく左右する。

---

## まとめ

MLOpsを「実験管理ツールを入れること」だと捉えると、本質を見失う。MLOpsの目的は、**機械学習モデルを継続的に、信頼できる状態で本番で動かし続ける仕組みを作ること**だ。

データ検証、再訓練パイプライン、モデル評価・承認、そして継続的監視。これらを連鎖させた自動化の仕組みが、「実験では良かったのに本番では…」という失敗パターンを防ぐ。

LLMの時代においても、この思想は変わらない。むしろ、モデルのブラックボックス化が進むからこそ、外から品質を測り続ける仕組みがより重要になる。
