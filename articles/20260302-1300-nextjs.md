# Next.js Middlewareの「正しい使いどころ」を考える

## はじめに

Next.jsのMiddlewareは便利な機能だ。リクエストがルートに到達する前に処理を挟み込み、リダイレクト、書き換え、レスポンスの返却ができる。

しかし「便利だから」という理由でなんでもMiddlewareに入れてしまうと、後々困ることになる。Middlewareには制約があり、その制約を理解していないと、思わぬパフォーマンス問題や動作の不安定さに悩まされる。

この記事では、Middlewareが何者であるか、何に向いていて何に向いていないかを整理する。

---

## Middlewareとは何か

Next.jsのMiddlewareは、Edge Runtimeで動作するコードだ。Node.jsのランタイムではなく、V8ベースの軽量ランタイムで実行される。

この「Edgeで動く」という特性が、Middlewareの使い方を決定的に規定する。

Edge Runtimeには次のような制約がある：
- Node.jsのAPIが使えない（`fs`, `path`, `crypto` の一部など）
- 利用できるnpmパッケージが限られる
- 実行時間に厳しい制限がある
- データベースへの直接接続は基本的に避けるべき

このため、Middlewareは「重い処理には向かない」。重い処理をMiddlewareに入れると、すべてのリクエストに対してその処理が走り、全体のレイテンシが悪化する。

---

## Middlewareが得意なこと

制約を踏まえると、Middlewareが本領を発揮するのはどんな場面か。

### 認証・認可のゲートキーパー

最も典型的な使い方は、リクエストに含まれるJWTやセッショントークンを検証し、未認証ユーザーをログインページにリダイレクトすることだ。

```ts
// middleware.ts
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const token = request.cookies.get('auth-token')?.value

  if (!token) {
    return NextResponse.redirect(new URL('/login', request.url))
  }

  // JWTの基本的な形式確認（Edge対応の軽量な検証）
  const isValidFormat = token.split('.').length === 3
  if (!isValidFormat) {
    return NextResponse.redirect(new URL('/login', request.url))
  }

  return NextResponse.next()
}

export const config = {
  matcher: ['/dashboard/:path*', '/settings/:path*'],
}
```

ここで重要なのは「軽量な検証」という点だ。JWTの署名検証を完全に行うことも技術的には可能だが、DBアクセスやネットワーク通信を伴う重い処理はMiddlewareには向かない。セッションの有効性の最終確認は、Server ComponentやAPI Routeに委ねる。

### A/Bテストのルーティング

Middlewareはリクエストのルーティングを変更できる。これを使って、ユーザーをバリアントAかバリアントBのページに振り分けるA/Bテストを実装できる。

クッキーでバリアントを固定し、同じユーザーが毎回同じバリアントを見るようにする——という仕組みをMiddlewareだけで実現できる。

### ジオロケーションによる地域別ルーティング

Vercelなどのプラットフォームでは、`request.geo`からリクエスト元の国情報を取得できる。これを使って、国ごとに異なるページや言語を提供するルーティングを実装できる。

### リクエストヘッダーの追加・変更

後続のServer ComponentやAPI Routeに追加情報を渡すためにヘッダーを付与することができる。ユーザーIDや権限情報をヘッダーに入れて渡すパターンだ。

---

## Middlewareに入れるべきでないもの

### データベースへのアクセス

繰り返しになるが、すべてのリクエストに対してDBアクセスが走ることは深刻なパフォーマンス問題を起こす。認証済みかどうかの確認を「必ずDBで」やろうとすると、Middlewareは適切な場所ではない。

### 重いビジネスロジック

ページアクセスのたびに複雑な計算や複数のAPI呼び出しが必要な処理は、Middlewareではなく各ルートのServer Componentや専用のAPIに置く。

### Middlewareでしかできない理由のない処理

「便利そうだからMiddlewareに入れる」は危険だ。理由なくMiddlewareに集中させると、あらゆるリクエストに対して不要な処理が走る。`matcher`で対象パスを絞ることもできるが、本当にMiddlewareである必要があるかを常に問い直す。

---

## matcherを活用して影響範囲を絞る

Middlewareのパフォーマンス問題を防ぐ重要な手段が `matcher` の設定だ。

```ts
export const config = {
  matcher: [
    // 静的ファイルとNext.js内部パスを除外
    '/((?!_next/static|_next/image|favicon.ico|.*\\.png$).*)',
  ],
}
```

デフォルトではすべてのリクエストにMiddlewareが適用される。静的ファイル（画像、CSS、フォントなど）に対しても毎回Middlewareが実行されると、無駄なオーバーヘッドが発生する。matcherで「本当にMiddlewareが必要なパス」だけに絞ることが基本だ。

---

## MiddlewareとServer Componentの役割分担

App RouterではServer Componentが強力なので、Middlewareとの役割分担に迷うことがある。

**Middlewareで行うこと：**
- リダイレクトとルーティング判断（ページの内容を返す前の段階）
- リクエストヘッダーの付与
- 静的・軽量な認証チェック

**Server Componentで行うこと：**
- 認証済みセッションの詳細検証
- データの取得と表示ロジック
- ユーザーごとの表示の出し分け

「誰がこのページを見られるか」はMiddlewareが判断し、「このユーザーに何を見せるか」はServer Componentが決める——という分担が自然だ。

---

## まとめ

Next.js Middlewareの「正しい使いどころ」は、**軽量でリクエスト横断的な処理**に絞られる。認証ゲート、ルーティング分岐、ヘッダー付与——これらはMiddlewareが真価を発揮する領域だ。

逆に、DBアクセスや重いビジネスロジックをMiddlewareに入れるのは、その設計思想に反する。Edge Runtimeの制約を「制限」として捉えるのではなく、「何に特化しているか」のシグナルとして読むことで、適切な設計判断ができるようになる。

便利な機能ほど、その特性をよく理解した上で使いたい。
