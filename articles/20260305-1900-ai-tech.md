## はじめに

「ファインチューニングをすれば、LLMが自社の知識を覚えてくれる」——そう思って試したことがある人は少なくないだろう。しかし実際にやってみると、期待通りの結果が得られないケースが多い。

ファインチューニングは「知識を追加する手術」ではない。より正確には、**モデルの出力傾向・スタイル・応答パターンを特定の方向に調整するプロセス**だ。この理解の違いが、成功と失敗を分ける。

本記事では、ファインチューニングの本質的な仕組みと、「何のためにやるのか」を整理する。

---

## ファインチューニングとは何をしているのか

事前学習済みのLLMは、膨大なテキストから「言語の構造」「知識」「推論パターン」を獲得している。ファインチューニングはそのモデルに対し、**特定のタスクやドメイン向けに追加学習**を行うことだ。

技術的には、ベースモデルのパラメータを（全部または一部）更新する。損失関数はベースとほぼ同じ（次のトークン予測）だが、使うデータがタスク特化のペアデータになる。

重要なのは、**ファインチューニングで「新しい知識」を大量に注入することは難しい**という点だ。パラメータを更新することで知識を上書きしようとすると、「catastrophic forgetting（破滅的忘却）」と呼ばれる現象が起き、以前の能力が劣化する。

---

## ファインチューニングが本当に得意なこと

ファインチューニングが真価を発揮するのは、以下のような用途だ。

### 1. スタイル・トーンの調整

「カジュアルな口調で答える」「箇条書きを使わず文章で返す」「社内用語を自然に使う」——こういった**出力スタイルの一貫化**はファインチューニングが非常に得意とする領域だ。

プロンプトで毎回指示するより、モデル自体がそのスタイルを「内在化」している状態の方が、精度が高く安定する。

### 2. 特定フォーマットへの出力変換

構造化されたJSON出力、特定のレポート形式、決まったテンプレートへの変換——こうした「形式への適応」もファインチューニングの強みだ。Few-shotだと例がコンテキストを圧迫するが、ファインチューニングならその必要がない。

### 3. 特定ドメインの推論パターン習得

法律文書の解釈、医療記録の要約、コードレビューのコメント生成など、**特定分野の思考パターン**を習得させるのに向いている。ただし「知識」ではなく「推論の仕方」を教える意識が重要だ。

## RAGとファインチューニングの使い分け

この2つはよく比較されるが、解決する問題が根本的に異なる。

| | RAG | ファインチューニング |
|---|---|---|
| 目的 | 最新・特定情報の参照 | 出力スタイル・タスク適応 |
| 知識更新 | 容易（データ追加のみ） | 再学習が必要 |
| コスト | 推論時コスト増 | 学習コスト |
| 向いている例 | 社内FAQ、製品情報 | コールセンター口調、特定フォーマット |

一般的な原則として、**「何を知っているか」はRAGで解決し、「どう振る舞うか」はファインチューニングで解決する**と考えるとわかりやすい。

多くの実用システムでは、この2つを組み合わせることが最も効果的だ。

## PEFT：全パラメータを更新しなくていい

近年はPEFT（Parameter-Efficient Fine-Tuning）と呼ばれる手法が主流になっている。代表的なのがLoRA（Low-Rank Adaptation）だ。

LoRAの発想はシンプルで、**元のモデルパラメータは凍結したまま、小さな「差分行列」だけを学習する**。この差分行列は元の行列と比べて圧倒的に小さく、学習コストが劇的に下がる。

```python
# LoRAの概念的なイメージ
# 元の重み行列W（変更しない）+ 学習する差分 ΔW = A × B
# A, Bは低ランク行列なので、パラメータ数が少ない
output = original_weight @ input + (lora_A @ lora_B) @ input
```

この差分部分だけを保存すれば、複数のLoRAアダプタを用途ごとに持てる。ベースモデルは共有しつつ、アダプタを差し替えるだけで振る舞いを切り替えられる——これがLoRAの実用的な強みだ。

## データ品質がすべてを決める

ファインチューニングで最も重要なのは、**学習データの品質**だ。「100件の高品質データ」は「10,000件の低品質データ」に勝ることが多い。

良いファインチューニングデータの特徴：

- **一貫性がある**：同じ状況に対して矛盾した応答が混在しない
- **ターゲットの振る舞いを体現している**：「こう答えてほしい」という理想の例が素直に入っている
- **エッジケースをカバーしている**：通常ケースだけでなく、難しいケースも含む
- **適切な量**：少なすぎれば学習不足、多すぎれば既存能力の劣化リスク

「とりあえず社内のQAデータを全部突っ込む」というアプローチが失敗しがちな原因は、**データの一貫性が保たれていない**ことがほとんどだ。

## 評価なきファインチューニングは迷走する

ファインチューニングを行う前に、**評価基準を明確に定義する**ことが不可欠だ。「改善した」をどう測るのか。

- 特定の出力フォーマット遵守率
- ドメイン特化タスクでの正答率
- 人間評価（A/Bテスト）

評価セットはファインチューニングデータと独立して準備する。学習に使ったデータで評価しても過学習を検出できない。

また、改善したいタスクで良くなった分、他のタスクが劣化していないか——これも必ずチェックする。ファインチューニングは局所最適化なので、必ず副作用を確認する姿勢が重要だ。

---

## まとめ：ファインチューニングを選ぶべき判断基準

ファインチューニングが適切な選択肢かどうかを判断するフレームワーク：

1. **プロンプトエンジニアリングで解決できるか？** → できるならまずそこから
2. **RAGで解決できるか？** → 知識参照が目的ならRAGが適切
3. **スタイル・形式・振る舞いの一貫化が目的か？** → ここがファインチューニングの本領
4. **十分な品質のデータを準備できるか？** → データなしには始められない
5. **評価基準が明確か？** → ここが曖昧なまま始めると迷走する

ファインチューニングは強力なツールだが、使いどころを間違えると費用と時間の無駄になる。「知識を詰め込む」ではなく「振る舞いを調整する」という本質を理解した上で使えば、LLMの可能性を大きく広げることができる。
