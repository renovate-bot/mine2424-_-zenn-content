## はじめに：フロントエンドとバックエンドの「型の断絶」

TypeScriptを使っているプロジェクトでも、フロントエンドとバックエンドの境界で「型の断絶」が起きることは珍しくない。

バックエンドがAPIのレスポンス型を変更した。でもフロントエンドはそれを知らず、古い型を前提にコードを書き続けた。そして本番環境でランタイムエラーが発生した——こういった経験は多くのエンジニアに覚えがあるだろう。

OpenAPIやGraphQLはこの問題へのアプローチを提供してきた。しかしtRPCはより軽量で、TypeScript自体の型推論を最大限活用するという異なる解を示した。本記事ではtRPCが何を解こうとしているか、そしてNext.jsとの組み合わせでどういう設計が可能かを探る。

## tRPCの核心：スキーマファイルを持たない型共有

tRPCが他のアプローチと根本的に異なるのは、スキーマ定義ファイルを別途持たないという点だ。

OpenAPIはYAML/JSONでAPI仕様を書き、それからクライアントコードを生成する。GraphQLはスキーマファイルを定義し、そこからTypeScriptの型を生成する。いずれも「スキーマを中心として、フロントとバックが独立して型を参照する」モデルだ。

tRPCの発想はシンプルかつ大胆だ。フロントエンドとバックエンドが同じTypeScriptプロジェクト（あるいはモノレポ）内にあるなら、バックエンドの型をそのまま直接インポートすればいい。スキーマファイルも、コード生成ステップも不要だ。

```typescript
// サーバー側: router定義
const userRouter = router({
  getUser: publicProcedure
    .input(z.object({ id: z.string() }))
    .query(async ({ input }) => {
      return await db.user.findUnique({ where: { id: input.id } });
    }),
});

// クライアント側: 型は自動的に推論される
const { data } = trpc.user.getUser.useQuery({ id: "123" });
// data の型は db.user.findUnique の返り値と一致する
```

`data` の型はバックエンドのリゾルバが返す型そのものだ。バックエンドが返す型を変えれば、クライアントの型エラーも即座に発生する。ビルド時にズレを検出できる。

## RouterとProcedureの設計

tRPCの基本構成要素はRouterとProcedureだ。

Procedureはエンドポイント一つに相当する。クエリ（読み取り）とミューテーション（書き込み）の2種類がある。各Procedureには入力スキーマ（Zodを使うのが一般的）と処理ロジックを定義する。

```typescript
const postRouter = router({
  list: publicProcedure
    .query(async () => {
      return await db.post.findMany({ orderBy: { createdAt: 'desc' } });
    }),

  create: protectedProcedure
    .input(z.object({
      title: z.string().min(1).max(100),
      content: z.string().min(1),
    }))
    .mutation(async ({ input, ctx }) => {
      return await db.post.create({
        data: { ...input, authorId: ctx.user.id },
      });
    }),
});
```

`publicProcedure` と `protectedProcedure` の区別が示すように、ミドルウェアを使って認証チェックを各Procedureに注入できる。これはAPIルート設計で頻繁に必要になる認証・認可ロジックを一元化する仕組みだ。

Routerは複数のRouterを入れ子にして構造化できる。`userRouter`、`postRouter`、`commentRouter` を一つの `appRouter` にまとめることで、APIの構造がコードの構造として表現される。

## Next.js App RouterとtRPCの組み合わせ

tRPCはReact Queryをベースにしたクライアントライブラリを提供しており、Next.jsとの統合はかなり洗練されている。

App Routerとの組み合わせでは、いくつかの設計選択肢がある。

一つ目は、Server Componentsから直接サーバー側のtRPCを呼び出すパターンだ。この場合、HTTPを経由せずに直接Procedureを呼べる。ネットワークオーバーヘッドなしで型安全なデータアクセスができる。

```typescript
// app/users/page.tsx (Server Component)
import { createCaller } from '@/server/trpc';
import { createContext } from '@/server/context';

export default async function UsersPage() {
  const ctx = await createContext();
  const caller = createCaller(ctx);
  const users = await caller.user.list();
  return <UserList users={users} />;
}
```

二つ目は、クライアントコンポーネントからReact Queryラッパーを通じてAPIを呼び出すパターンだ。インタラクティブな操作やリアルタイム更新が必要な場面で使う。

この2パターンの使い分けが、App RouterとtRPCを組み合わせる際の設計判断の核心になる。

## Zodとの連携：入力バリデーションの統合

tRPCはZodと非常に自然に統合される。ProcedureのInputにZodスキーマを渡すと、入力バリデーションが自動的に実行され、バリデーションエラーは適切なHTTPエラーとして返される。

この設計の利点は、入力バリデーションのロジックを一か所に集約できることだ。フロントエンドでも同じZodスキーマを再利用すれば（共通パッケージとして切り出すか、直接インポートして）、クライアントサイドバリデーションとサーバーサイドバリデーションで同一のルールを保証できる。

## tRPCが向かないケース

tRPCはすべての状況に最適なわけではない。正直に向かないケースも挙げておく。

まず、外部クライアントへのAPI公開には向かない。モバイルアプリや外部パートナーにAPIを提供する場合、tRPCのTypeScript依存は制約になる。このケースでは、OpenAPIによる標準的なREST APIやGraphQLの方が適切だ。

次に、マイクロサービス間通信も注意が必要だ。tRPCはTypeScriptプロジェクト内での型共有を前提としており、異なるサービスを跨ぐ場合の型安全性は別途担保が必要になる。

また、チームがTypeScriptに不慣れな場合、tRPCの型推論の仕組みを理解するまでの学習コストがかかる。型エラーのメッセージが複雑になることもある。

## バンドル分離とSSRの注意点

tRPCクライアントはクライアントサイドのJavaScriptバンドルに含まれる。App Routerを使う場合、クライアントコンポーネントとサーバーコンポーネントの境界を意識した設計が必要だ。

サーバーコンポーネントでは直接Callerを使い、クライアントコンポーネントでのみtRPCクライアントを使う——このルールを守ることで、バンドルサイズを必要以上に増やさずに済む。

tRPCはHTTP Batchingもサポートしており、複数のクエリを一つのHTTPリクエストにまとめて送れる。これはウォーターフォール問題を軽減する上で効果的だ。

## おわりに：型安全性の価値をどこまで追求するか

tRPCが示したのは、「型安全性をランタイムではなくコンパイル時に確保する」という方向性だ。

ランタイムで型エラーを検出するより、ビルド時に検出した方がコストが低い。フロントエンドとバックエンドの境界での型の断絶は、TypeScriptの型推論を活用することで最小化できる。

これは「正しい問いを立てること」でもある。APIの型安全性をどこで、どうやって確保するか——この問いへの答えが、tRPCなのかOpenAPIなのかGraphQLなのかを決める。プロジェクトの規模、チームの構成、外部公開の有無。これらを踏まえた上で選択することが、設計者としての判断だ。
