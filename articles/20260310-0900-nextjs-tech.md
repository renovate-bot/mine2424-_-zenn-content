## はじめに

Next.jsアプリのテストを書こうとしたとき、最初に直面するのは「何をテストすべきか」という問いだ。コンポーネントのDOMをテストするのか、ユーザーの操作フローをテストするのか、Server Componentsはどう扱うのか——ツールを揃えるより先に、テスト戦略の設計思想を理解する必要がある。

## テストピラミッドからの出発

テスト設計の基礎はテストピラミッドだ。底辺にユニットテスト、中間に統合テスト、頂点にE2Eテストが位置する。コストと信頼性のバランスから、ユニットを厚く・E2Eを薄くするのが原則だ。

ただしReactコンポーネントのテストは少し異なる見方が必要だ。「1つのコンポーネントだけをテストする」という従来のユニットテストの発想より、**ユーザーの視点で機能のまとまりをテストする**という考え方が現代では主流になっている。これがReact Testing Library（RTL）の設計哲学だ。

## React Testing Libraryの核心思想

RTLの作者Kent C. Doddsは「テストコードはソフトウェアの使われ方に似ていればいるほど信頼できる」と述べている。

この思想が具体化されるのが、クエリの選び方だ。RTLは `getByRole`・`getByLabelText`・`getByText` を優先し、`getByTestId` を最後の手段とする。

なぜか。`getByRole('button', { name: '送信' })` は、ユーザーが「送信ボタン」を探すときの認知と同じプロセスをたどる。`getByTestId('submit-btn')` は実装の詳細に過ぎず、ユーザーには見えない。

テストコードがユーザーの視点に近いほど、UIリファクタリング（内部実装の変更）でテストが壊れにくくなる。

## Next.js App Routerでのテスト設計

App RouterではServer ComponentsとClient Componentsが混在する。これがテスト設計に影響する。

**Client Componentsは通常通りRTLでテストできる**。`'use client'` ディレクティブがあるコンポーネントはブラウザ環境で動くため、Jest + RTLの組み合わせがそのまま使える。

**Server Componentsは統合テストの対象**というのが現実的な判断だ。Server ComponentsはNode.js環境でしか動作せず、Jestでレンダリングするには複雑な設定が必要になる。完全にテストするより、Playwrightなどを使ったE2Eテストや統合テストで動作を確認するほうが保守コストが低い。

重要なのは**ビジネスロジックをServer ComponentsのなかにネストさせないComposeパターン**だ。データ取得・変換ロジックは独立した関数として切り出し、その関数をユニットテストする。コンポーネント自体のテストはUIの振る舞いに絞る。

## 何をテストし、何をテストしないか

設計の本質は「テストしない判断」にある。

**テストする価値が高いもの**:
- ユーザーインタラクション（クリック・入力・送信）の結果
- 条件分岐によるUI変化（ローディング状態・エラー表示・空状態）
- フォームバリデーションのフィードバック
- アクセシビリティ属性（role・aria-label）が正しく設定されているか

**テストしなくていいもの**:
- コンポーネントのpropsをそのまま表示するだけのもの
- CSSクラス名・スタイルの詳細
- サードパーティライブラリの内部動作
- 実装の詳細（useState・useEffectの発火タイミングなど）

「どう実装されているか」ではなく「ユーザーに何が届くか」をテストする。

## ユーザーイベントのテスト

RTLでは `userEvent` を使ったインタラクションテストが重要だ。

```typescript
import userEvent from '@testing-library/user-event'

it('フォーム送信でサクセスメッセージが表示される', async () => {
  const user = userEvent.setup()
  render(<ContactForm />)
  
  await user.type(screen.getByLabelText('メールアドレス'), 'test@example.com')
  await user.type(screen.getByLabelText('メッセージ'), 'テストメッセージ')
  await user.click(screen.getByRole('button', { name: '送信' }))
  
  expect(await screen.findByText('送信が完了しました')).toBeInTheDocument()
})
```

`fireEvent` より `userEvent` が推奨される理由は、実際のブラウザのイベント発火順序（mousedown→mouseup→click）をシミュレートするからだ。

## テストファイルの構成とカバレッジ

Next.jsのテストファイルはコンポーネントと同じディレクトリに置く（`__tests__/` ディレクトリ or `.test.tsx` ファイル）のが標準だ。

カバレッジ数値を追いかけることには注意が必要だ。80%のカバレッジを達成することが目標になった瞬間、テストの質が下がる。コードの行数をカバーすることと、ユーザーに価値を届けることは別の問いだ。

## まとめ

Next.jsのテスト戦略の核心は「ユーザーの視点でテストする」ことだ。RTLのクエリAPI・userEventはその思想を実現するための道具に過ぎない。

何をテストするかを決めるのはツールではなく設計だ。Server Componentsのビジネスロジックを関数に切り出し、Client Componentsのインタラクションをユーザー視点でテストする——この構造を持てば、テストはリファクタリングに強く、バグを本当に防ぐものになる。
