## 本番環境のエラーはどこに消えるのか

開発環境ではエラーがすぐに目の前に現れる。コンソールに赤いログが出て、スタックトレースが見えて、問題の場所がわかる。しかし本番環境では状況が違う。ユーザーがエラーに遭遇しても、それはサーバーのログの奥に埋もれるか、そもそも記録されないまま消えていく。

「本番でだけ起きるエラー」は珍しくない。異なる認証状態、本番だけのデータパターン、特定のブラウザやOSの組み合わせ、負荷がかかったときだけ発生する非同期の問題——こうした問題を後から追跡しようとすると、「いつ」「どこで」「誰が」「どんな状況で」という情報がなければ再現も修正も難しい。

Sentryはこの問題を解決するためのエラートラッキングサービスだ。アプリケーションで発生したエラーを収集し、スタックトレース、ユーザーの操作履歴、デバイス情報、コンテキストをまとめてダッシュボードに表示する。本記事では、Next.js App RouterへのSentry統合と、有効に使うための設計の考え方を整理する。

## Next.js App RouterでSentryが難しくなった理由

Pages RouterではSentryの統合が比較的シンプルだった。クライアントサイドのエラーは`_app.tsx`で、サーバーサイドのエラーは`_error.tsx`でまとめてハンドリングできた。

App Routerでは構造が変わった。Server Components、Client Components、Route Handlers、Server Actions——それぞれが異なるランタイムコンテキストで動く。Server Componentsのエラーはサーバー側でのみ発生し、クライアントには伝わらない。Server Actionsのエラーはフォーム送信の中で発生する。エラーの発生場所が分散しているため、一箇所で全部を捕まえるアプローチが機能しなくなった。

Sentry for Next.js（`@sentry/nextjs`パッケージ）は、このApp Routerの構造に対応した統合を提供している。ただし自動的に取れるエラーと、明示的に設定が必要なエラーがある。

## セットアップの考え方

`npx @sentry/wizard@latest -i nextjs`コマンドが設定の多くを自動化する。以下のファイルが生成される。

- `sentry.client.config.ts`: クライアントサイドのSentry設定
- `sentry.server.config.ts`: サーバーサイドのSentry設定
- `sentry.edge.config.ts`: Edge Runtimeの設定
- `instrumentation.ts`: Next.jsのOpenTelemetry互換の計装エントリポイント

`instrumentation.ts`はNext.js 13.2以降で導入されたファイルで、サーバーの初期化時に一度だけ実行される。ここでSentryを初期化することで、App Routerのサーバーサイドエラーを確実にキャプチャできる。

```typescript
// instrumentation.ts
export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    await import('./sentry.server.config');
  }
  if (process.env.NEXT_RUNTIME === 'edge') {
    await import('./sentry.edge.config');
  }
}
```

## サンプリングレートの設計

Sentryに全イベントを送ると、高トラフィックのアプリでは膨大なデータ量と料金になる。`tracesSampleRate`（パフォーマンストレースのサンプリング）と`sampleRate`（エラーのサンプリング）を環境に応じて設定する。

```typescript
// sentry.client.config.ts
import * as Sentry from "@sentry/nextjs";

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  tracesSampleRate: process.env.NODE_ENV === "production" ? 0.1 : 1.0,
  // 本番では10%のトレースを送信、開発では全件
  replaysSessionSampleRate: 0.1,
  replaysOnErrorSampleRate: 1.0,
  // エラー発生時のセッションリプレイは全件
});
```

エラー自体は原則として全件送信し、パフォーマンストレースはサンプリングするという設計が一般的だ。Replays（ユーザーの操作録画）はエラー発生時のみ全件送信し、通常セッションはサンプリングすることでコストを抑えられる。

## エラーバウンダリとの統合

App RouterのServer Componentでエラーが発生すると、同じルートセグメントの`error.tsx`がフォールバックUIとして表示される。しかし`error.tsx`のデフォルト動作ではSentryに自動でエラーが送信されない。

`error.tsx`でエラーを受け取り、明示的にSentryに送信する設計が必要だ。

```tsx
// app/error.tsx
"use client";
import * as Sentry from "@sentry/nextjs";
import { useEffect } from "react";

export default function ErrorPage({ error }: { error: Error & { digest?: string } }) {
  useEffect(() => {
    Sentry.captureException(error);
  }, [error]);

  return (
    <div>
      <h2>問題が発生しました</h2>
      <p>しばらく待ってから再度お試しください。</p>
    </div>
  );
}
```

`digest`はNext.jsがサーバーサイドのエラーに付ける識別子だ。クライアントには詳細なエラー情報を渡さず（セキュリティ上の理由から）、digestをSentryのログと照合することでサーバーログと紐づけられる。

## コンテキストを豊かにする

エラーが発生したという情報だけでは再現が難しい。「誰が」「何をしていたとき」「どんなデータで」という文脈をエラーに付加することで、調査コストが大幅に下がる。

**ユーザーコンテキストの設定**

認証済みユーザーの情報（匿名化したIDなど）をSentryに設定することで、エラーが特定のユーザーセグメントに集中していないかを確認できる。

```typescript
Sentry.setUser({
  id: user.id,
  // メールアドレスなど個人情報は慎重に扱う
  // GDPRなどのプライバシー規制との整合を確認する
});
```

**カスタムタグとエクストラ情報**

エラーにタグを付けることで、フィルタリングと集計が容易になる。「どのAPIエンドポイントで失敗したか」「どのフィーチャーフラグが有効だったか」といった情報をタグとして追加する。

## ソースマップとエラーの読みやすさ

本番のJavaScriptはminify・bundleされているため、スタックトレースが難読化されている。ソースマップをSentryにアップロードすることで、元のソースコードに対応したスタックトレースを表示できる。

`@sentry/nextjs`はビルド時にソースマップを自動でアップロードする設定を提供している。`SENTRY_AUTH_TOKEN`を環境変数に設定すれば、`next build`のタイミングで自動アップロードが行われる。ソースマップを公開したくない場合（セキュリティ上の理由など）は、アップロード後にデプロイされたファイルから削除する設定も可能だ。

## アラートのノイズを管理する

Sentryを導入したばかりのとき、膨大なアラートが来て疲弊するチームが多い。まず全エラーをリストアップし、「対処が必要なエラー」と「無視してよいエラー」を分類する作業から始めるのが現実的だ。

サードパーティのブラウザ拡張が引き起こすエラー、ネットワーク断絶による`fetch`の失敗、ボットのクローリングによるエラーは、多くの場合無視してよい。Sentryのフィルタリング機能（`beforeSend`）で特定のエラーパターンを除外することで、本当に対処すべきエラーが見えやすくなる。

「エラーレート」の変化をアラート基準にすることも有効だ。絶対数ではなく「1時間で100件以上急増した」というような条件にすれば、デプロイ後の回帰をすばやく検知できる。

## まとめ

SentryをNext.js App Routerに統合することで、本番環境の「見えないエラー」を可視化できる。Server Components、Client Components、Route Handlersそれぞれで発生するエラーを確実にキャプチャするには、`instrumentation.ts`の設定と`error.tsx`での明示的な送信が必要だ。ソースマップのアップロード、ユーザーコンテキストの設定、アラートのノイズ管理を整えることで、Sentryはエラーの通知ツールから「本番問題の調査基盤」へと昇格する。
